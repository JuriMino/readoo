つまづき①：ログイン画面が 500 エラー

エラー Missing required parameter for [Route: password.reset] ... [Missing parameter: token]

導線リンクを route('password.reset') にしていたのが原因。password.reset は reset-password/{token} でtoken パラメータが必須のルート。route() はBlade 描画時に評価されるため、リンクを置いたログイン画面そのものが表示できなくなっていた。

修正： 申請フォームへ飛ばす route('password.request')（＝forgot-password）に変更。

学び： 名前付きルートでパラメータ必須のもの（{token} や {id}）を、パラメータ無しで route() 呼びするとその場で例外。しかも描画時評価なので「リンクを置いただけのページ」が巻き添えで落ちる。導線の route() は「今パラメータを渡せるか」を意識する。

つまづき②：リセット確定時に SQL エラー

エラー Unknown column 'remember_token' in 'field list'

リセット処理自体は最後まで動いたが、NewPasswordController が更新時に必ず remember_token を再発行しようとして失敗。v1 で「remember me はスコープ外」として users テーブルから rememberToken() を除外していたため、カラムが存在しなかった。

修正方針： カラムを追加し直す（＝設計判断を覆す＋migrate:fresh）のは過剰。remember_token はリセットに必須ではないので、コントローラー側で書き込み行をコメントアウトして対応した。

$user->forceFill([
    'password' => Hash::make($request->password),
    // 'remember_token' => Str::random(60),   // remember me はv1スコープ外のため無効化
])->save();

2. 開発用メール確認環境（Mailpit）

「実際にメールが飛んで見た目を確認したい」ため、送信ドライバを検討。本番用サービスと開発用キャッチャーを整理した。

選択肢位置づけ今回の判断
Mailpitローカルのメールキャッチャー（Dockerコンテナ）採用（開発用）
Mailtrap外部SaaSのサンドボックス受信箱見送り（登録が必要）
Resend本番向け送信サービス（無料枠＋独自ドメイン認証）デプロイ時に採用予定
log ドライバメール本文をログに書き出すだけ初期の動作確認に使用
Mailpit を選んだ理由： すでに Docker Compose 構成なのでサービスを1つ足すだけで完結し、アカウント登録も外部送信も不要。受信メールは http://localhost:8025 の画面で HTML プレビュー付きで確認できる。

compose.yaml に追加

  mailpit:
    image: axllent/mailpit:latest
    ports:
      - "8025:8025"   # 確認用のWeb画面
      - "1025:1025"   # メール受信ポート（SMTP）

src/.env（開発）

MAIL_MAILER=smtp
MAIL_HOST=mailpit     # Dockerのサービス名で名前解決
MAIL_PORT=1025
MAIL_USERNAME=null
MAIL_PASSWORD=null
本番の残タスク： デプロイ時に Resend で独自ドメインを認証（DKIM/SPF の DNS レコード追加）し、MAIL_MAILER を smtp(Mailpit) から resend に切り替えるだけ。.env は gitignore 対象なので、環境ごとに値を持てる。

4. プロフィールのパスワード変更バグ修正 issue #12 / PR #13

症状 プロフィール編集でパスワードを変更し Update しても、エラーが出ないのに保存されない。ログアウト後、新パスワードでログイン失敗・古いパスワードでログイン成功（＝DBが更新されていない）。

切り分け：二重ハッシュを疑ったが違った

コントローラーは Hash::make() で明示ハッシュし、User モデルには 'password' => 'hashed' キャストもある。二重ハッシュを疑い、DB を汚さず tinker で検証した。

$u = new App\Models\User();
$u->password = Hash::make("test1234");
Hash::check("test1234", $u->password);  // => true（単一ハッシュ）

結果は true。hashed キャストはすでにハッシュ済みの値なら再ハッシュしない（内部で Hash::isHashed() で判定）ため、二重ハッシュは起きていなかった。＝これは原因ではない。

真因：フォームの input 名とコントローラーの期待キーの不一致

PasswordController::update が検証・取得するキーは Breeze 標準の password / password_confirmation。一方、日本語化したフォームの name は独自の new_password / confirm_password だった。

フォームの nameコントローラーの期待
current_passwordcurrent_password一致
new_passwordpassword不一致
confirm_passwordpassword_confirmation不一致

そのため送信データに password が存在せずバリデーションで停止し、更新処理まで到達していなかった（＝古いパスワードのまま）。

エラーが画面に出なかった理由： エラーは updatePassword という名前付きバッグの password キーに入るが、新パスワード欄の表示は ->get('new_password') という存在しないキーを見ていたため、どこにも描画されなかった。

学び： 「無反応で保存されない」はバリデーション失敗＋エラー表示キーのズレの合わせ技で起きやすい。フォームの name は、それを受けるコントローラー／FormRequest のルールキーと必ず一対一で合わせる。エラー表示（$errors->bag->get('キー')）も同じキーで揃える。

修正

update-password-form.blade.php の新パスワード欄・確認欄の name / id / label for / エラー表示キーを、Breeze 標準の password / password_confirmation に統一。（current_password と autocomplete は元々正しいので変更なし）

次フェーズの方針（明日以降）

① UI/UX の仕上げ・フロントエンド調整

機能は揃ったので、使用感の観点で修正したい点を洗い出し、UI 仕上げに入る。まずは「気になる点の棚卸し」から。観点の例：

  • 画面ごとの余白・タイポグラフィ・色使いの一貫性（ボタン・カード・バッジの統一）
  • 空状態（データ0件）の表示、ローディング／完了フィードバック
  • フォームのエラー表示・必須表現の分かりやすさ
  • 導線・戻り先の自然さ、モバイル幅でのレイアウト崩れ
  • 操作後のトースト等、結果が伝わるフィードバック

② 一覧画面の検索・絞り込み（v1で追加検討）

使用感として、一覧画面での検索・絞り込みは v1 時点で欲しいと判断。追加実装を検討する。対象は Book／Knowledge／Action の各一覧。想定する切り口：

  • キーワード検索（タイトル・著者・内容 等）
  • 絞り込み（例：Book のステータス／ジャンル、Knowledge・Action の参照元の本 など）
  • 件数が増えた際の並び替え・ページネーションも併せて要検討

実装前に、各一覧で「何で探せると嬉しいか」を先に定義してから、コントローラーのクエリ（when() による条件付き絞り込み）とフォームUIに落とす想定。
