書籍管理（books） ルート設計 提案

Readoo! — Book CRUD のルート定義方針

対象画面と要件

画面HTTP / URL
書籍一覧GET /books
書籍登録GET /books/create ／ POST /books
書籍詳細GET /books/{book}
書籍編集GET /books/{book}/edit ／ PUT・PATCH /books/{book}
書籍削除DELETE /books/{book}
  • 認証必須（未ログインはアクセス不可）
  • 自分が登録した本にのみアクセス可（詳細・編集・削除）
  • ルート名は books.index のようなLaravel標準命名

結論：Route::resource を採用推奨

理由：books は教科書的な7アクションCRUDであり、仕様の「RESTful 7アクション」と完全一致する。 1行で標準命名のルートが揃い、DRYで保守性が高い。個別並べ書きの柔軟性は今回不要。

1. ルート定義コード（写経用）

既存の Route::middleware('auth')->group()（認証グループ）の中に1行追加する。

use App\Http\Controllers\BookController;   // ファイル冒頭の use 群に追加

Route::middleware('auth')->group(function () {
    // 既存の profile ルート …

    Route::resource('books', BookController::class);
});
verified ミドルウェアは付けない：メール認証はv1スコープ外。profile ルートも auth のみで揃えているため、 books も auth だけに統一する。

個別に書いた場合の等価コード（＝resourceが裏でやっていること）

Route::get('books',             [BookController::class, 'index'])->name('books.index');
Route::get('books/create',      [BookController::class, 'create'])->name('books.create');
Route::post('books',            [BookController::class, 'store'])->name('books.store');
Route::get('books/{book}',      [BookController::class, 'show'])->name('books.show');
Route::get('books/{book}/edit', [BookController::class, 'edit'])->name('books.edit');
Route::put('books/{book}',      [BookController::class, 'update'])->name('books.update');
Route::delete('books/{book}',   [BookController::class, 'destroy'])->name('books.destroy');

2. なぜこの書き方か

ルート名（books.index 等）

Route::resource('books', ...) はリソース名 books を接頭辞に、 books.index / create / store / show / edit / update / destroy を自動生成する。 手書きの ->name() 不要でブレない。

ミドルウェア：認証はルート、認可はPolicy

重要な切り分け：
  • 認証（ログインの有無）＝auth ミドルウェア。グループで囲い全7ルートに一括適用。
  • 認可（自分の本だけ操作可）＝Policy。ルートでは表現できない。 /books/{book} の {book} がルートモデルバインディングで Book に解決され、 BookPolicy が「このユーザーの所有物か？」を判定する（Controller実装時）。

合言葉：ログインの有無＝認証＝ミドルウェア / 他人のデータを触らせない＝認可＝Policy。

resource vs 個別並べ書き

観点Route::resource（推奨）個別並べ書き
規約適合RESTful 7アクションと完全一致自分で7行書く
命名標準名を自動生成（ブレない）手書き（タイプミスの余地）
可読性・保守1行で意図が明確行数が増える
向くケース標準CRUD（今回）変則URL・一部アクションのみ・独自名

将来一部だけにしたくなったら ->only([...]) / ->except([...]) で絞れる（今回は7つ全部使うので不要）。

3. Controller に実装する7メソッド

メソッドHTTP / URLルート名役割引数
index()GET /booksbooks.index一覧表示なし
create()GET /books/createbooks.create登録フォームなし
store()POST /booksbooks.store登録処理FormRequest
show(Book $book)GET /books/{book}books.show詳細表示Book
edit(Book $book)GET /books/{book}/editbooks.edit編集フォームBook
update(Request, Book $book)PUT/PATCH /books/{book}books.update更新処理FormRequest + Book
destroy(Book $book)DELETE /books/{book}books.destroy削除処理Book
ルートモデルバインディング：URLパラメータ名 {book} とメソッド引数の型ヒント Book $book が対応すると、 IDから自動で Book を取得（Book::find() 手書き不要）。
雛形生成（後で）：php artisan make:controller BookController --resource --model=Book で7メソッドの雛形が作れる（今は提案のみ・実行不要）。

確認コマンド

docker compose exec app php artisan route:list --path=books

7ルート（index/create/store/show/edit/update/destroy）が想定どおり生成されるか確認できる。

このあとの全体の流れ（見通し）

  1. Book モデル＋マイグレーション（user_id 外部キー ← 所有者判定の土台）
  2. Route::resource 追加 ← 今回の提案
  3. BookController（7メソッド。ロジックは BookService へ分離）
  4. BookPolicy（所有者のみ show/edit/update/destroy 可）
  5. FormRequest（バリデーション）
  6. Blade（一覧・詳細・フォーム。エンティティ色は緑＝book）
  7. Feature テスト
