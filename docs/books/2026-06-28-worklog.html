① 退会時の削除設計（物理 / 論理の使い分け）

「削除」と一口に言っても2種類ある。Readoo!では操作の意味で使い分けることにした。

操作対象削除方法理由
本単体の削除（詳細🗑️）その本論理削除 $book->delete()誤操作の復旧余地を残す
退会ユーザー論理削除 $user->delete()退会記録を残す
退会本・知識・行動物理削除 forceDelete()完全削除（個人データを残さない）

スキーマ構造（知識・行動は「本の子」）

User ──(user_id)──▶ Book ──(book_id)──▶ Knowledge ──(knowledge_id)──▶ Action
                          └─(book_id)──────────────────────────────▶ Action

knowledges.book_id / actions.book_id / actions.knowledge_id はいずれも cascadeOnDelete()。つまり本を物理削除すると、DBが知識・行動を自動で物理削除する。

public function destroy(Request $request): RedirectResponse
{
    $request->validateWithBag('userDeletion', [
        'password' => ['required', 'current_password'],
    ]);

    $user = $request->user();
    Auth::logout();

    DB::transaction(function () use ($user) {
        // 本を物理削除 → 知識・行動はカスケード制約で自動的に物理削除される
        $user->books()->withTrashed()->forceDelete();

        // メールアドレスを退避して元アドレスを解放（同じメールで再登録できるように）
        $user->email = $user->email . '_deleted_' . $user->id;
        $user->save();

        // ユーザー本体は論理削除（記録を残す）
        $user->delete();
    });

    $request->session()->invalidate();
    $request->session()->regenerateToken();

    return Redirect::to('/');
}
記述意味
withTrashed()->forceDelete()論理削除済みの本も含めて行ごと完全削除。これが知識・行動のカスケード削除のトリガー
知識・行動の削除コードが無いDBの cascadeOnDelete() が自動処理（＆モデル未実装なので書けない）。将来モデルを作っても退会処理は変更不要
DB::transaction(...)「本は消えたがユーザー処理で失敗」等の中途半端を防ぐ。エラー時は全部巻き戻る（オール・オア・ナッシング）
子→親の順外部キー制約があるため、子（books）を先に消してから親（user）を処理

② メールアドレスの再登録対応（MySQL特有の事情）

ユーザーを論理削除にすると、users.email は unique() 制約のためレコードが残り、 同じメールで再登録できない問題が出る。

MySQLでは「部分ユニークインデックス」が使えない。 PostgreSQLなら「deleted_at IS NULL の行だけにユニーク」できるが、MySQLは不可。 また「(email, deleted_at) の複合ユニーク」も、MySQLはNULLを互いに別物として扱うため、 現役ユーザー（deleted_at = NULL）同士で同じメールが登録できてしまい一意性が壊れる。→ この手は使えない。
採用：退会時にメールを退避して元アドレスを解放。 aaa@example.com → aaa@example.com_deleted_5 に書き換える。 元アドレスが「未使用」になるので、登録側の標準ルール unique:User::class のまま再登録が通る。
  • 退会者は SoftDeletes でそもそも認証時に取得されない（ログイン不可）→ メールを変えても無害
  • _deleted_5（id）付与で退避後メールが絶対に衝突しないことを保証
  • 接尾辞を外せば元アドレスは復元可能＝記録は保持

③ Featureテスト一式（計20本）

テストは SQLiteメモリDB 上で動く（本番MySQLに触れない・使い捨て・高速）。 既存のBreezeテストと同じ PHPUnitスタイル＋RefreshDatabase で統一。

準備：BookFactory ＋ HasFactory

  • Bookモデルに use HasFactory; を追加（無いと Book::factory() が使えない）
  • BookFactory 新規作成。'user_id' => User::factory() で持ち主も自動生成
FactoryはなぜFillableに無い user_id を入れられるのか： Factoryは内部で Model::unguarded() によりmass assignment保護を一時的に外すため。 本番コード（コントローラ）では保護が効くので、アプリの安全性はそのまま。

テスト内訳

ファイル本数主な確認
BookTest9一覧は自分の本だけ／登録（user_idが自分）／バリデーション／詳細・編集・更新・論理削除／未ログインはlogin
BookAuthorizationTest4他人の本は show/edit/update/destroy すべて403。かつデータが変化しないことまで確認
ProfileTest（追加2）2退会で本は物理削除（論理削除済みも）／同じメールで再登録できる
テストの勘どころ：
  • assertForbidden()（403）＝認可エラー。未ログインの「login へリダイレクト」＝認証エラー。両者は別物
  • 403を返すだけでなく assertDatabaseHas / assertNotSoftDeleted で副作用が無いことまで検証
  • 退会の物理削除は assertDatabaseMissing（行ごと消滅）、論理削除は assertSoftDeleted（行は残る）で区別
  • assertDatabaseMissing/Has は生テーブルを直接見るため、'deleted_at' => null を条件に足して「現役ユーザーだけ」に絞る
  • 'user_id' => $user->id のassertはmass assignment防御の確認（リクエストに含めていないのに自分のIDが入る）

④ 今日詰まった点（テストのタイポ）

テストメソッド名のタイポが連発したが、いずれも即 BadMethodCallException で気づけた （Tailwindのクラス名タイポが「黙って無視」されるのと対照的）。

症状原因修正
Method ... does not existassertDontSeee（e×3）assertDontSee
undefined method ...assertSessionHasNotErrorsassertSessionHasNoErrors
スタックトレースの読み方： エラー表示の at .../ForwardsCalls.php:67 は「例外をthrowしたフレームワーク内部の行」。 未定義メソッドは __call 経由で ForwardsCalls に転送され、そこでthrowされるため毎回67行目になる。 自分が直すべき行（例：ProfileTest.php:128）はスタックトレースを下に辿ると出てくる。 php artisan test --filter=... -v で全トレースを表示できる。
