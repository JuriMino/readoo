① ルート定義

use App\Http\Controllers\BookController;

Route::middleware('auth')->group(function () {
    // 既存の profile ルート …
    Route::resource('books', BookController::class);
});

1行で7ルート（index/create/store/show/edit/update/destroy）が標準命名（books.index 等）で生成される。 詳細は docs/books/books-routes-proposal.html 参照。

採用理由：books は教科書的な7アクションCRUDで仕様の「RESTful 7アクション」と完全一致するため Route::resource を採用。 個別並べ書きより DRY で保守性が高い。verified ミドルウェアはメール認証がv1スコープ外なので付けず auth のみ。

③ BookPolicy（所有者制限）＜詳しく＞

そもそも「ポリシー（Policy）」とは何か

Policy ＝「あるモデルに対して、このユーザーは何をしてよいか」を判定する“認可ルール”の専用クラス。

Readoo! の要件「自分が登録した本にしかアクセスできない（詳細・編集・削除）」を実現する中心。 BookPolicy は Book 専用で、操作ごとにメソッド（view / update / delete …）を持ち、 各メソッドが true（許可）/ false（拒否） を返す。

なぜ Policy が必要か（認証では足りない）

見るもの担当
認証（authentication）ログインしているか？auth ミドルウェア
認可（authorization）ログイン済みの上でこの本を操作してよいか？Policy

auth は「ログインの有無」しか見ないので、「この本があなたのものか」は判定できない。そこを Policy が担う。 合言葉：ログインの有無＝認証＝ミドルウェア／他人のデータを触らせない＝認可＝Policy。

なぜ Controller に直接書かず Policy に分けるのか

  • 一元管理：所有者ルールが1か所に集約（各メソッドにコピペが散らばらない）
  • Fat Controller回避：規約どおり認可ロジックをコントローラから追い出せる
  • 再利用：Controller（authorizeResource）でも Blade（@can）でも同じルールを使える

今回の中身：書き換えたもの / 書き換えなかったもの

php artisan make:policy で生成した雛形は全メソッドが return false（＝全部禁止）。 そこから必要なメソッドだけ中身を変えた。「$book を受け取るかどうか」で書き方が3パターンに分かれるのがポイント。

メソッド対応アクション$book対応中身
viewAnyindex（一覧）なし書き換えreturn true;
viewshow（詳細）あり書き換え所有者チェック
createcreate/storeなし書き換えreturn true;
updateedit/updateあり書き換え所有者チェック
deletedestroy（削除）あり書き換え所有者チェック
restore（v1未使用）ありそのままreturn false;
forceDelete（v1未使用）ありそのままreturn false;

パターンA：所有者チェックにした（view / update / delete）

public function update(User $user, Book $book): bool
{
    // ログイン中ユーザーのidと、本の持ち主(user_id)が一致するか
    return $user->id === $book->user_id;
}

理由：この3つは特定の本（$book）を対象とする操作。 引数で $book を受け取れるので「その本の user_id ＝ ログインユーザーか」を判定できる。 これが要件「自分の本だけ詳細・編集・削除できる」の本体。

パターンB：true にした（viewAny / create）

public function viewAny(User $user): bool
{
    return true;
}

理由：この2つは特定の本が存在しない操作。

  • viewAny（一覧）：対象は「本の集合」で個別の $book が無い → 所有者判定のしようがない。 「自分の本だけ表示」はPolicyではなくコントローラのクエリ（auth()->user()->books）で実現する。 Policyは「一覧ページにアクセスしてよいか」だけ見るので true。
  • create（登録）：これから作る本なのでまだ持ち主が存在しない → 所有者チェック不可。ログイン済みなら誰でも作成可なので true。
ここを false のままにすると壊れる：雛形の初期値 false のままだと、 authorizeResource 経由で 一覧(index)と登録(create/store)が403になる。必ず true に変える。

パターンC：false のまま（restore / forceDelete）

理由：v1には「復元（restore）」機能が無く、forceDelete（退会時の物理削除）は コード内で直接呼ぶもので Policy 経由ではない。どのルートからも到達しないので、 デフォルトの false（安全側＝拒否）のままでよい。

未使用 import の補足：雛形が付ける use Illuminate\Auth\Access\Response; は今回未使用（bool で返しているため）。 残してもエラーにならないが削除可。Response::deny('理由') のように拒否理由メッセージを返したいときに使うもの。

このPolicyがどう呼ばれるか（次回 Controller で）

// BookController の中で1行書くと、各アクションに対応するPolicyが自動チェックされる
$this->authorizeResource(Book::class, 'book');
//   show → view / update → update / destroy → delete が自動適用、他人の本は403

Bladeでは @can('update', $book) … @endcan で「自分の本のときだけ編集ボタン表示」も可能。

★ その他の押さえておくべきポイント

① user_id は Fillable に入れない（セキュリティ）

持ち主はサーバー側で $book->user_id = auth()->id() として設定する。 Fillableに入れるとリクエスト改ざんで他人の user_id で本を作れてしまう（マスアサインメント脆弱性）。 Policyの所有者判定が信頼できるのは、user_id を入力させていないからこそ。

② SoftDeletes と退会時の物理削除（落とし穴）
  • 通常の本削除＝論理削除（SoftDeletesトレイト）
  • 退会時は紐づく本を物理削除する想定だが、FKの cascadeOnDelete() は ユーザーが物理削除されたときだけ発火。退会＝ユーザーの論理削除なのでcascadeは効かない。
  • → 退会処理で $user->books()->withTrashed()->forceDelete() を明示的に呼ぶ必要がある（将来実装）。
③ ルートモデルバインディング

URLの {book} とコントローラ引数 Book $book が対応すると、IDから自動で Book を取得（find() 不要）。 この $book がそのまま Policy に渡って所有者判定される。

④ casts は手動と自動を区別

手動で書くのは started_at / finished_at（date型）だけ。 created_at / updated_at と deleted_at は自動でcastされるので書かない。

疑問に思ったところ（Q&A）

Q. Book に Hidden は不要？ casts は date項目だけ？

A. その通り。Hidden は password 等の秘匿情報用で Book には不要。casts は started_at/finished_at を date。 cast するとCarbonになり ->format('Y/m/d') 等が使える。

Q. BookPolicy とは？

A. 「この本にこのユーザーが何をしてよいか」を判定する認可ルールの置き場所。 要件「自分の本だけ操作可」を $user->id === $book->user_id で表現。 Controllerで authorizeResource、Bladeで @can として使う。認可ロジックを集約してFat Controllerを防ぐ。（詳細は上の③）

Q. update以外のPolicyメソッドも同じ書き換え？

A. 同じではない。$book を持つ view/delete は update と同じ所有者チェック。 $book を持たない viewAny/create は true。restore/forceDelete は v1未使用で false のまま。（上の3パターン参照）
