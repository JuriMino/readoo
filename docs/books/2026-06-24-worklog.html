① 認可の組み込み方 ― なぜ authorizeResource を使わなかったか

重要な判明事項：当初 authorizeResource() をコンストラクタで使う予定だったが、 Laravel 11以降では動かない。ベースの Controller.php が空クラスになり、 middleware() メソッドを持たなくなったため。authorizeResource() は内部で $this->middleware() を呼ぶので、呼ぶと Call to undefined method middleware() で落ちる。
方法書き方評価
A. 各メソッドで $this->authorize()（採用）メソッド冒頭に1行ずつ認可が目に見える・学習向き・確実
B. HasMiddleware + can: ミドルウェアstatic middleware() を実装自動だが記述が冗長

採用にあたり、空のベースControllerには $this->authorize() が無いので、 BookController に use AuthorizesRequests; トレイトを追加した。

use Illuminate\Foundation\Auth\Access\AuthorizesRequests;

class BookController extends Controller
{
    use AuthorizesRequests; // $this->authorize() が使えるようになる

authorize の2つの渡し方（重要）

場面書き方理由
一覧・作成（対象の本がまだ無い）authorize('viewAny', Book::class)クラス名を渡す
詳細・編集・更新・削除（特定の本）authorize('view', $book)インスタンスを渡す（所有者比較のため）

② index ― 一覧の本当の防御は「クエリの絞り込み」

public function index()
{
    $this->authorize('viewAny', Book::class);

    /** @var \App\Models\User $user */
    $user = Auth::user();
    $books = $user->books()->latest()->get();

    return view('books.index', ['books' => $books]);
}
押さえどころ：viewAny は return true なので「ログインしていれば通る」関門にすぎない。 一覧で他人の本が漏れないのは $user->books() の時点で WHERE user_id = 自分のID が自動で付くから。 一覧の所有者保護はPolicyではなく「リレーション経由のクエリ絞り込み」が担う、という役割分担を理解すること。

latest() ＝ orderBy('created_at', 'desc') の短縮（新しい順）。

IDE警告の話：auth()->user() / Auth::user() に出る赤線

これは実行エラーではなく、IDE（Intelephense）の誤検知。 auth() の戻り値は Factory インターフェース型で user() を持たない、と判定されるため。 Auth::user() に変えても今度は戻り値 Authenticatable に books() が無いと言われ、警告場所がズレるだけ。

根本原因は「戻り値がインターフェース型で、IDEが具象 App\Models\User だと分からない」こと。 変数に型を教える1行で解決する（実行には無影響のコメント）：

/** @var \App\Models\User $user */
$user = Auth::user();
$books = $user->books()->latest()->get();

auth() と Auth:: は機能同じ（好みでよい）。Auth:: 使用時は use Illuminate\Support\Facades\Auth; が必要。 根本的に消すなら barryvdh/laravel-ide-helper 導入も手だが、今回は @var で十分。

③ store ― user_id を守る「3重の防御」（最重要）

public function store(BookRequest $request)
{
    $this->authorize('create', Book::class);

    $validated = $request->validated(); // BookRequestで検証済み

    /** @var \App\Models\User $user */
    $user = Auth::user();
    $user->books()->create($validated); // user_id を自動セット

    return redirect()->route('books.index')->with('status', '書籍を登録しました');
}

「登録する本の持ち主は必ず本人」を、3つの仕組みが重なって保証している：

  1. $validated に user_id が入らない … validated() は検証ルールに書いたキーだけを返す。フォームに <input name="user_id"> を偽装されても検証対象外なので無視される。
  2. Bookの #[Fillable] に user_id を入れていない … 仮に混入してもFillable外は弾かれる（mass assignment対策）。
  3. $user->books()->create() が user_id = 自分のID を自動付与 … hasManyリレーション経由で作るので、自分で $book->user_id = ... と書く必要がない。

疑問：最後の with('status', ...) はbooksテーブルのstatusカラム？

→ 全くの別物。たまたま同じ単語なだけ。
場所意味値の例
booksの status カラムDB / $validated['status']本の読書状態unread/reading/finished
with('status', ...)セッションのフラッシュデータ画面に出すお知らせ文「書籍を登録しました」

with('status', ...) はリダイレクト先に1回だけ渡すフラッシュメッセージ（DBは触らない）。 キー名が 'status' なのは、Breeze生成ビューが session('status') を読む作りに合わせているため （プロフィール更新でも同じ仕組みを使用）。紛らわしければ別名にもできるが、既存と統一する方が楽。

④ show / edit / update / destroy ― 認可の本番

ルートモデルバインディング（string $id → Book $book）

引数の型を Book にするだけで、Laravelが /books/5 の 5 から Book::findOrFail(5) を自動実行して $book に入れてくれる。
  • 自分で Book::find($id) を書く必要がない
  • 該当IDが無ければ自動で404
  • SoftDeletesのおかげで、論理削除済みの本も自動404（URL直打ちされても安全）
public function show(Book $book)    { $this->authorize('view', $book);   return view('books.show', ['book' => $book]); }
public function edit(Book $book)    { $this->authorize('update', $book); return view('books.edit', ['book' => $book]); }

public function update(BookRequest $request, Book $book)
{
    $this->authorize('update', $book);
    $book->update($request->validated());
    return redirect()->route('books.show', $book)->with('status', '書籍を更新しました');
}

public function destroy(Book $book)
{
    $this->authorize('delete', $book);
    $book->delete(); // SoftDeletesなので deleted_at が入るだけ（物理削除しない）
    return redirect()->route('books.index')->with('status', '書籍を削除しました');
}
認可失敗は「403」が正解（認証の「ログイン画面リダイレクト」と混同しない）： 他人の本のURL（例 /books/3）を直打ちすると authorize('view', $book) が $user->id === $book->user_id で弾き、403 Forbidden を返す。 「未ログイン→ログイン画面へリダイレクト」は認証（authミドルウェア）の話で、別レイヤー。 ログイン済みだが権限が無い＝認可＝403。

能力名の対応関係

Policyに edit メソッドは作らない。「編集フォームを開ける＝更新できる権限がある人」なので、 edit も update 能力で判定する。同様に create↔store、view↔show。

⑤ BookRequest（FormRequestでバリデーション共通化）

生成コマンド：docker compose exec app php artisan make:request BookRequest

class BookRequest extends FormRequest
{
    public function authorize(): bool
    {
        return true; // 認可はController側のPolicy($this->authorize)で実施
    }

    public function rules(): array
    {
        return [
            'title'       => ['required', 'string', 'max:255'],
            'author'      => ['required', 'string', 'max:100'],
            'publisher'   => ['nullable', 'string', 'max:100'],
            'status'      => ['required', 'in:unread,reading,finished'],
            'genre'       => ['required', 'in:business,self_help,technology,...,other'],
            'started_at'  => ['nullable', 'date'],
            'finished_at' => ['nullable', 'date', 'after_or_equal:started_at'],
            'summary'     => ['nullable', 'string'],
            'memo'        => ['nullable', 'string'],
        ];
    }

    public function attributes(): array // エラー文の項目名を日本語に
    {
        return ['title' => 'タイトル', 'author' => '著者', /* … */];
    }
}

Controller側は引数を Request → BookRequest に変え、ルール配列を削除して $request->validated() を使う。これで store / update の重複が消える（DRY）。

FormRequestの押さえどころ：
  • 検証タイミング：メソッドの中身が動く前に自動で検証 → NGなら自動リダイレクト＆エラー表示。コントローラ内は「検証済み」が保証される。
  • $request->validated()：検証を通過したキーだけを返す。all() と違いルール外（偽装 user_id 等）は入らない → これもmass assignment対策の一部。
  • authorize(): bool：今回は所有者チェックをController側に集約したので true（素通し）。生成直後の値が false だと全リクエスト403になるので注意。
  • after_or_equal:started_at：読了日が開始日より前にならない相関チェック（両方入力時のみ作動）。
