★ 重点復習①：autofocus

「ページを開いた瞬間にカーソルが入る欄」を1つだけ指定する属性。見た目ではなく“入力開始位置”。

  • ページ読み込み時に、その入力欄へ自動でカーソルが入る（クリック不要で入力開始できる）。
  • 1ページに1つだけ有効。複数付けても最初の1つしか効かない。
  • 付けるのは「最初に入力してほしい欄」。ログイン=email、登録=username。
今日のミス：register画面で username と email の両方に autofocusが付いていた。 → email側を削除。最初に入力すべき username だけに残すのが正しい。

★ 重点復習②：autocomplete

入力欄の「意味」をブラウザに伝える裏方ラベル。補完候補やパスワード管理を“正しい種類で”動かすための交通整理。

大前提：候補が出る仕組みは2系統ある

種類何が候補に出るか担当
① フォーム履歴（オートフィル）過去にそのブラウザで入力した値（メール等）ブラウザ本体
② パスワードマネージャー保存済みのログイン情報（ID・パスワード）ブラウザ / 1Password 等

「入力途中で過去の値が候補に出た」のは主に①フォーム履歴（ブラウザの基本機能）。 autocomplete はこの①②を正しく・的確に動かすための指示であって、候補を出す魔法そのものではない。

よく使う値

値意味使う場所
usernameログインIDになる欄（メールでも該当すれば使う）ログイン/登録のメール欄
emailログインには使わない連絡先メール純粋なメール入力欄
current-password既存のパスワードログインのパスワード欄
new-passwordこれから作るパスワード登録・パスワード変更欄
off補完しない（※パスワード欄では効かないことも）補完されたくない欄
最重要の使い分け：current-password と new-password
  • ログイン＝current-password：既存パスワードを呼び出して入力してほしい
  • 新規登録＝new-password：強力なパスワードを生成・提案してほしい
  • 登録の「パスワード」と「パスワード確認」は両方 new-password → 同じ生成パスワードが両欄に入る
今日のミス：register のパスワード欄が current-password（ログイン用）のままだった → new-password に修正。

username と email の使い分け

判断軸は「ログインIDとして使うか」。username は“ユーザー名”ではなく“ログインID”を表すトークン。 メールアドレスでも、それでログインするなら username を使う。

Readoo! はメールでログインする仕様（＝メール＝ログインID）。 なのでログイン・登録ともにメール欄は autocomplete="username" で揃えると、パスワードマネージャーが 「ID＋パスワード」をセットで保存・自動入力でき、次回ログインが繋がりやすい。

表示用の「ユーザー名（username欄）」はログインIDではないので、ここに autocomplete="username" を付ける必要はない（混同しない）。

確かめ方：メール欄に打ち始めて過去値が出れば①フォーム履歴。パスワード欄クリックで「強力なパスワードを生成」が出れば②マネージャー（new-passwordが効いている合図）。

★ 重点復習③：input の type 属性の罠

今日の実害バグ：パスワード確認欄を type="password_confirmation" と書いていた。
type="password_confirmation" というtypeは存在しない → 不正なtypeは type="text" 扱いになる → 確認欄がマスクされず平文で丸見えになっていた。

正：type="password"（マスク）＋ name="password_confirmation"（送信名）。 type（入力の種類）と name（送信フィールド名）は役割が別物。混同しないこと。

あわせて、確認欄のラベルが for="password" になっていた（正しくは for="password_confirmation"）。 for は対応する入力の id と一致させる。ズレるとラベルクリックで別の欄にフォーカスが飛ぶ。

★ 重点復習④：auth()->check() と Auth::check()

どちらも同じ動作。Auth::check() はファサード（要 use インポート）、auth()->check() はヘルパ関数。

  • IDE/静的解析が「ファサードの方が明示的・解析しやすい」として Auth::check() を推奨することがある（今日の警告の正体）。実害のある間違いではない。
  • use Illuminate\Support\Facades\Auth; を冒頭に書けば Auth::check() が使える。
  • プロジェクト内で書き方を統一しておくと読みやすい（今後 Auth:: で揃える方針）。

ルート設定の変更（/ の入口）

routes/web.php の / を、welcomeページ表示からログイン状態の分岐に変更。

Route::get('/', function () {
    return Auth::check()
        ? redirect()->route('dashboard')   // ログイン済み → ダッシュボード
        : redirect()->route('login');      // 未ログイン → ログイン画面
});
  • view() ではなく redirect() を採用：既存の名前付きルート（login / dashboard）を再利用でき、URLも統一される。
  • welcome.blade.php は参照されなくなるだけで害はない。後で整理時に削除でOK。
  • 反映確認：docker compose exec app php artisan route:clear

今日の学びの一言まとめ

  • autofocus＝最初にカーソルを置く欄を1つだけ。
  • autocomplete＝欄の意味を教える裏方。パスワードは ログイン=current / 登録=new。メールログインなら ID欄は username。
  • type と name は別物。type=入力の種類、name=送信名。
  • for は id と一致させる。
  • Auth::check() と auth()->check() は同じ。ファサードで統一。
