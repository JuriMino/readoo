1. テストの基本方針（Book をミラー）

既存の BookTest / BookAuthorizationTest と同じ2ファイル構成を踏襲。知識ならではのポイントは 「Knowledge は Book 経由の間接所有」である点。

親子の組み立て：Book::factory()->for($user) → Knowledge::factory()->for($book)

User → Book → Knowledge の親子関係を明示的に作る。->for() は belongsTo（book() / user()）を辿って所有者を固定するので、「自分の本に紐づく知識」「他人の本に紐づく知識」を作り分けられる。

DB は phpunit.xml で SQLite の :memory:。RefreshDatabase で毎テスト初期化されるため、実DBを汚さず高速に回る。

テスト確認内容
index自分の本に紐づく知識だけ表示、他人のは出ない
create画面200 で表示できる
store登録され book_id が自分の本／既定は本の詳細へリダイレクト

編集画面で参照元の本を付け替えられない設計（update() の except('book_id')）を検証。将来うっかり book_id を fillable に足しても、このテストが落ちて気づけるという「回帰の防波堤」になる。

検証したい項目以外は「正常値」を送る

requires title テストでは title を省く一方、book_id と content は有効値を送る。book_id を空にすると book_id のエラーも同時に出て、「title が原因で落ちた」ことが濁ってしまう。1テスト1関心を保つコツ。

3. KnowledgeAuthorizationTest.php（認可・5本）

テスト確認している防御
view others他人の知識詳細は 403
edit others他人の編集画面は 403
update others403 ＋ 内容が変わっていない
delete others403 ＋ 削除されていない
create on others book他人の本IDでの作成は バリデーションで拒否（session エラー）＋ 未登録
なぜ「作成」だけ 403 ではないのか

既存データ（view/edit/update/delete）は「本 → user_id」を Policy で辿れるので 403。
一方 作成時はまだ知識が存在しないため Policy で守れない。代わりに「その book_id は自分の本か？」を入力値のバリデーションで検証している：
Rule::exists('books','id')->where('user_id', $this->user()->id)
他人の本IDは「自分の本の中には存在しない」扱いになり、book_id のバリデーションエラー（redirect＋session エラー）になる。＝ 403 ではなく 422相当の差し戻し。この層の違いを理解しておくと、認可の設計が読みやすくなる。

403 の確認は「返り値」だけでなく「データ保護」まで

update / delete の他人テストは、assertForbidden() に加えて assertDatabaseHas(...元の値) / assertNotSoftDeleted() まで見る。「403 は返るが実は書き換わっていた」を防ぐため、状態が変わっていないことまでアサートする。

4. 写経ミスとテストの効用

store（from=index）が失敗 → 原因はテスト側の from → tag1

実際のリダイレクト先が books/1（本の詳細）になり期待の knowledges と不一致。テストのペイロードで 'from' => 'index' が 'tag1' => 'index' に化けていたため、from が届かず既定分岐になっていた（＝コードは正しい）。tag1 を from に直して解決。

教訓：もし from 分岐に本当のバグがあっても、tag1 を送っていたら「たまたま通る」ことはない。期待した経路を URL レベルで検証できている証拠でもあった。

Tips：テスト実行コマンド

自分のターミナルで回す場合（対話ターミナルなので -T は不要。-T は自動実行側で疑似TTYを割り当てないためのオプション）。

基本

# 全テスト実行（コミット前にこれを1回流す習慣を）
docker compose exec app php artisan test

絞り込み

# ファイル単位（クラス名）
docker compose exec app php artisan test --filter=KnowledgeTest

# メソッド単位（1本だけ）
docker compose exec app php artisan test --filter=test_knowledge_can_be_updated

# スイート単位
docker compose exec app php artisan test --testsuite=Feature

便利オプション

# 最初の失敗で止める（デバッグ時に速い）
docker compose exec app php artisan test --stop-on-failure

# 並列実行で高速化（テスト数が増えたとき）
docker compose exec app php artisan test --parallel
補足
  • チャットに結果を出したいときは、プロンプトで ! を付けて実行：! docker compose exec app php artisan test
  • コンテナが止まっていたら先に docker compose up -d。
  • 素の PHPUnit（./vendor/bin/phpunit）でも動くが、artisan test の方が出力が見やすい。
