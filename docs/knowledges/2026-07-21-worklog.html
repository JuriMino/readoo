② 削除設計の整理：知識削除は行動を巻き込まない

本の削除では、紐づく知識・行動は（一覧のフィルタにより）ユーザー目線で「一緒に消えた」扱いになる。 しかし知識の場合、行動の必須の親は本（book_id必須）であり、知識はあくまで任意リンク （actions.knowledge_idはnullable）。したがって知識を削除しても、 行動は本に紐づく行動として残すのが設計と一貫する、という結論になった。

実装上もこの結論と一致していた： KnowledgeController::destroyは$knowledge->delete()（ソフトデリート）であり、 knowledge_id外部キーのcascadeOnDelete()制約は物理削除時のみ発火するため、 通常の削除操作ではそもそも発火しない（これは退会時の全物理削除カスケード用の制約）。 つまり「行動は消えない」は追加実装ではなく既存の実挙動の言語化だった。

削除確認モーダルの文言修正

書籍の削除モーダルの注意書き「紐づく知識や行動も削除されます」をそのまま流用すると事実と異なるため、 知識の削除モーダルは以下に変更。

<div class="mt-4 flex items-start gap-2 text-sm text-red-600">
    <span aria-hidden="true">⚠️</span>
    <p>この知識に紐づく行動は削除されません<br>（本に紐づく行動として残ります）</p>
</div>
アイコンと2行テキストのバランス調整： アイコンと本文を1つのflex items-center行に入れて<br>で改行すると、 アイコンがブロック全体の縦中央（1〜2行目の間）に浮いてしまう。 items-startに変え、アイコンを1行目の高さに揃えることで解消した。

③ Featureテストで挙動を固定

「知識削除で行動は消えない」という設計判断は、将来カスケード設定が変わった時などに気づけるよう テストで固定化しておく価値がある。KnowledgeTestに1本追加。

// 知識を削除しても、紐づく行動は削除されない（本に紐づく行動として残る）
public function test_deleting_knowledge_does_not_delete_linked_actions(): void
{
    $user = User::factory()->create();
    $book = Book::factory()->for($user)->create();
    $knowledge = Knowledge::factory()->for($book)->create();
    $action = Action::factory()->for($book)->create(['knowledge_id' => $knowledge->id]);

    $this->actingAs($user)
        ->delete(route('knowledges.destroy', $knowledge))
        ->assertRedirect(route('books.show', $book));

    $this->assertSoftDeleted($knowledge);
    $this->assertDatabaseHas('actions', [
        'id'         => $action->id,
        'deleted_at' => null,
    ]);
}
レビューでの指摘：初回実装時にメソッド名がtest_deleting_knowledge_dose_not_...とタイポしていた。 PHPUnitはメソッド名の意味を解釈しないため実行・PASSはするが、可読性のためdoes_notに修正。

④ 見つかった次の課題：行動編集での知識の付け替え

知識から行動を作成したあと、その知識を削除すると、行動は本に残り続けるが「知識と紐づけ直す手段がない」 ことに気づいた。ActionController::updateは現状book_id・knowledge_idを 両方とも更新対象から除外しており（「参照元の付け替えはしない想定」）、知識を削除できる仕様にする以上、 知識側だけ編集不可のままなのは片手落ちになる。

方針案（明日以降の行動画面修正で対応予定）：
  • 本（book_id）は行動の必須の親なので引き続き編集不可のまま
  • 知識（knowledge_id）は任意リンクなので、同じ本に属する知識から選び直せるように編集画面へセレクトを追加
  • ActionController::updateのexceptからknowledge_idを外し、store()と同様に検証済みの時だけセットする
