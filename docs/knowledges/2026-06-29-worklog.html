① Knowledgeモデルとリレーション

知識は user_id を持たず book_id で本にぶら下がる。所有者は本を経由して辿る構造。

User ──(user_id)──▶ Book ──(book_id)──▶ Knowledge
  • #[Fillable(['title','book_page','content','tag1','tag2','tag3'])] … book_id は除外（mass assignment防御。書籍の user_id と同じ思想）
  • use HasFactory, SoftDeletes;
  • book(): BelongsTo ／ Book側に knowledges(): HasMany（HasMany のuse追加も必要）
今日の罠：不可算名詞モデルのテーブル名。 Knowledge は英語で不可算名詞のため、Laravelの複数形化が knowledge（変化なし）になる。 実テーブルは knowledges なので食い違い、「テーブルが無い」エラーになる。
→ 実機で (new Knowledge)->getTable() が knowledge を返すことを確認 → モデルに protected $table = 'knowledges'; を明示して解消。
information / equipment / media 等も同じ罠。

② ルート設計（独立一覧 ＋ shallow nested）

「一覧」は2種類あり、性質が違うので分けて設計した。

一覧の種類表示場所ルート
特定の本に絞った一覧本の詳細画面の中（「紐づく知識」テーブル）専用ルート不要（books.show 内で $book->knowledges を描画）
本をまたいだ独立一覧専用ページGET /knowledges（knowledges.index）
// 独立した知識一覧（本をまたいで自分の全知識）
Route::get('knowledges', [KnowledgeController::class, 'index'])->name('knowledges.index');

// 作成・登録は本の中（ネスト）、詳細以降は知識単体（フラット）
Route::resource('books.knowledges', KnowledgeController::class)
    ->shallow()
    ->except(['index']);
アクションURLルート名
index（独立一覧）knowledgesknowledges.index
createbooks/{book}/knowledges/createbooks.knowledges.create
storebooks/{book}/knowledges (POST)books.knowledges.store
showknowledges/{knowledge}knowledges.show
editknowledges/{knowledge}/editknowledges.edit
updateknowledges/{knowledge} (PUT)knowledges.update
destroyknowledges/{knowledge} (DELETE)knowledges.destroy
shallow() の意図：「作成・登録」はどの本かが必要なのでネスト、「詳細・編集・削除」は知識IDだけで足りるのでフラット。 前半ネスト・後半フラットを自動でやってくれる。本ごとの一覧は本詳細が兼ねるので except(['index']) でネスト版indexは生やさない。

③ KnowledgePolicy（間接オーナー判定）

書籍は book->user_id で直接判定できたが、知識は本を経由して持ち主を辿る。

// 詳細・更新・削除：本経由でオーナー判定
public function view(User $user, Knowledge $knowledge): bool
{
    return $user->id === $knowledge->book?->user_id;
}

// 作成：その本の持ち主だけが追加できる（本を受け取る）
public function create(User $user, Book $book): bool
{
    return $user->id === $book->user_id;
}
記述意味
$knowledge->book?->user_id本経由でオーナーを辿る。?->（ヌル安全）で、本が取得できない場合は null → $user->id === null ＝ false で安全に拒否（403）
create(User $user, Book $book)「作れるか」はどの本に追加するかで決まるので本を受け取る。通常の create(User) と形が違う
コントローラ側の呼び方$this->authorize('create', [Knowledge::class, $book])。第1要素＝Policy解決用クラス、第2要素以降＝メソッド引数
Policy登録Laravel 11+ は Knowledge → KnowledgePolicy を命名規則で自動発見。手動登録不要

⑤ KnowledgeController（叩き台・明日写経）

叩き台を提示済み。書籍コントローラと同型だが、ネストと間接認可が加わる。要点だけ記録。

  • use AuthorizesRequests; 必須（空のベースControllerに authorize() が無いため）
  • index（独立一覧）：Knowledge::whereHas('book', fn($q) => $q->where('user_id', $user->id))->with('book')->latest()->paginate(15) … 知識は user_id を持たないので「自分の本を持つ知識」を whereHas で絞る。with('book') でN+1回避。 （別解：User に hasManyThrough を定義すれば $user->knowledges() でも可）
  • create / store：ネストルートから Book $book を受け取り、authorize('create', [Knowledge::class, $book])。 登録は $book->knowledges()->create($request->validated()) で book_id 自動セット
  • show / edit / update / destroy：Knowledge $knowledge（ルートモデルバインディング）。各冒頭で authorize
  • destroy：リダイレクト先が本詳細なので $book = $knowledge->book; を削除前に控えてから $knowledge->delete()（論理削除）
  • リダイレクト：store/destroy → books.show、update → knowledges.show
確認の注意：この時点ではビュー（knowledges.*）が無いので画面アクセスはエラー。 写経後は「php artisan route:list が通る／構文エラーが無い」までの確認でOK（画面確認はタスク6以降）。
