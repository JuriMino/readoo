2. 写経で見つかったバグ・修正

【動作バグ】ドロップダウンの初期選択が効かない（=== → ==）

当初 @selected(old('book_id', $selectedBookId) === $book->id) と厳密比較にしていた。しかし：

  • $book->id は整数（例 3）
  • old('book_id') / $selectedBookId はリクエスト由来なので文字列（例 '3'）

'3' === 3 は false になり、初期選択も差し戻し時の再選択も効かない。→ 緩い比較 == に修正して解決。

なぜ books/create では === で通っていたか

books/create の @selected(old('genre') === $value) は GENRES のキーが文字列同士だったため === で成立していた。今回は「整数 vs 文字列」なので緩い比較が必要、という違い。

3. キャンセルボタンの遷移分岐

「キャンセル後の遷移先が必ず独立した知識一覧になる」問題を修正。Cancel は <a> リンクなので、フォーム送信ではなく描画時に from を読んで分岐する。

<a href="{{ request('from') === 'book' && $selectedBookId
    ? route('books.show', $selectedBookId)
    : route('knowledges.index') }}">Cancel</a>
来た画面from / book_idCancel 遷移先
本の詳細（?from=book&book_id=3）book / ありbooks.show（元の本）
独立一覧（?from=index）index / なしknowledges.index
直接 /knowledges/createなしknowledges.index（既定）
差し戻し後もクエリが維持される

入力エラーで store → create フォームへ戻るとき、遷移先は元の GET URL（?from=book&book_id=3 付き）。そのため request('from') / $selectedBookId は再度読める。
→ 「登録後のリダイレクト」も「キャンセル」も、同じ from ルールで来た画面に戻るで揃った（登録後はコントローラ側、キャンセルはビュー側と場所は違うが判定基準は同一）。

4. 本の詳細との連携

BookController@show（eager load）

$book->load(['knowledges' => fn ($query) => $query->latest()]);
クロージャで並び順を指定

単に $book->load('knowledges') でも読めるが、新しい順で表示したいのでリレーション読み込みに latest() を差し込むクロージャを渡す。ビューでのループ時の N+1 も防げる。

books/show.blade.php に「この本から得た知識」セクション追加

  • ヘッダー右に 「+ 知識を追加」ボタン → route('knowledges.create', ['from' => 'book', 'book_id' => $book->id])
  • book_id で作成フォームの本を初期選択、from=book で登録/キャンセル後この詳細へ戻す
  • 空状態（「まだ知識が登録されていません」）＋ 知識リスト（タイトル・該当箇所・タグ）
  • タグは一覧と同じ array_filter([tag1, tag2, tag3]) で null を除去して描画
