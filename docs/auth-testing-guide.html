認証機能テスト 手順書

Readoo! — Feature テスト（登録・ログイン・プロフィール・退会）

このドキュメントのゴール

認証機能（新規登録・ログイン・ログアウト・プロフィール編集・退会）が正しく動くことを、 自動テスト（Feature テスト）で検証する。Breeze生成テストはカスタマイズ前提とズレているため、 まず土台を直してから実行する。

前提知識

Unit テストと Feature テストの違い

種類検証対象場所
Unit テスト1つのクラス・メソッドを単独でtests/Unit/
Feature テストHTTPリクエスト→画面遷移・DBまで通しでtests/Feature/

今回のチェック項目（登録できる・ログインできる等）は通しの動作確認なので Feature テスト。

テスト環境

項目設定意味
フレームワークPHPUnitテスト実行ツール
テスト用DBSQLite（メモリ上）本番MySQLとは別・使い捨て・高速。本番データに触れない
RefreshDatabase各テストでDBリセットテスト間でデータが混ざらない
メールarrayドライバ実送信しない
安心ポイント：テストは SQLite メモリDB上で動くので、本番・開発のMySQLデータには一切影響しない。何度でも実行してよい。

テストの土台を直す（最重要・これをしないと必ず失敗）

なぜ必要か：Breeze生成のファクトリ／テストは「nameカラム・メール認証あり・物理削除」前提。 本プロジェクトは「username・メール認証なし・論理削除」にカスタマイズ済みなので、 実装が正しくてもテストが失敗する。先にテスト側を実装に合わせる。

① database/factories/UserFactory.php を修正

存在しないカラム（name / email_verified_at / remember_token）を入れようとして落ちる。

public function definition(): array
{
    return [
        'username' => fake()->name(),
        'email' => fake()->unique()->safeEmail(),
        'password' => static::$password ??= Hash::make('password'),
    ];
}

削除：email_verified_at 行 ／ remember_token 行 ／ unverified() メソッド

② スコープ外のテストファイルを削除

ファイル理由
tests/Feature/Auth/EmailVerificationTest.phpメール認証はv1スコープ外
tests/Feature/Auth/PasswordResetTest.phpパスワードリセットはルート無効化済み
tests/Feature/ExampleTest.php/ が200前提だが今は302リダイレクト

※ PasswordConfirmationTest.php は使わない機能。消さなくても通る可能性あるが、気になれば削除。

③ RegistrationTest.php を修正

POSTデータの 'name' → 'username' に変更。

④ ProfileTest.php を修正（3点）

  • 'name' → 'username'、$user->name → $user->username
  • email_verified_at を使う行、および test_email_verification_status... メソッドを丸ごと削除
  • 退会テストを論理削除対応に（下記）
// test_user_can_delete_their_account の末尾
// 変更前： $this->assertNull($user->fresh());
$this->assertSoftDeleted($user);
論理削除の落とし穴：SoftDeletes では退会後もレコードが残る（deleted_at が入るだけ）。 fresh() はスコープを無視して削除済みレコードも拾うため assertNull だと失敗する。 論理削除専用の assertSoftDeleted() を使う。
1

テストを実行する

# 全テスト
docker compose exec app php artisan test

# 特定ファイルだけ
docker compose exec app php artisan test tests/Feature/ProfileTest.php

# テスト名で絞り込み（メソッド名の一部でOK）
docker compose exec app php artisan test --filter=test_new_users_can_register

発展：自分でテストを1本書いてみる

「未ログインで /profile にアクセスすると login へリダイレクトされる」を検証するテスト。 ProfileTest.php に追加してみると練習になる。

public function test_guest_is_redirected_to_login(): void
{
    $response = $this->get('/profile');

    $response->assertRedirect('/login');
}

ポイント：ログインしていない状態（actingAs を呼ばない）で保護ページにアクセスし、 assertRedirect('/login') でログイン画面へ飛ばされることを確認する（仕様：未ログインはリダイレクト）。

完了チェックリスト

  • UserFactory を username / email / password だけに修正した
  • EmailVerificationTest / PasswordResetTest / ExampleTest を削除した
  • RegistrationTest の name → username を直した
  • ProfileTest の name → username、email_verified_at 削除、退会を assertSoftDeleted に直した
  • php artisan test で全テストが緑になった
  • （発展）未ログインのリダイレクトテストを自分で追加した
