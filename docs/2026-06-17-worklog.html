確定したデザイントークン

緑・赤・グレーはTailwind標準を使い、パレットに無いブランド系のみ定義した（tailwind.config.js）。

book #16a34a本（緑）
knowledge #2D5BE3知識（青）=secondary
action #E8510A行動（橙）=primary
base #FAF6EC背景クリーム

色のルール（決定事項）

  • エンティティ色：本=緑 / 知識=青 / 行動=オレンジ。行動＝メインカラーはコアコンセプト（行動化）に最重要要素を当てる設計意図。
  • 操作色（固定）：削除=赤(red-600) / 戻る・キャンセル=グレー(gray-500)。エンティティに関係なく固定。
  • 緑・赤・グレーはTailwind標準でOK（追加定義不要）。
tailwind.config.js の構成メモ： 色コードの重複を避けるため const brand = {green, blue, orange} を export default の外（importの下）に定義し、 colors から brand.orange のように参照する形にした。const は extend の中には書けない（オブジェクト内はキー:値のみ）点に注意。

設計上のポイント（なぜそうしたか）

  • 見出し「Log in」はカード外＝login.blade.php側に配置。guest layoutに置くとregister画面でも「Log in」と出てしまうため、画面ごとのbladeに持たせた。
  • Remember meチェックボックスは削除。v1スコープ外（migrationで rememberToken をコメントアウト済み）かつモックにも無いため。
  • ナビ・背景・フッターは guest layout に集約 → login と register の両方に1回で適用される。
  • フッターの年は date('Y') で動的化。
  • レイアウトは min-h-screen flex flex-col ＋ <main> を flex-1 にして、フッターを常に最下部へ。
見た目を変えても消してはいけない要素（重要）： @csrf ／ name="email"・name="password" ／ <x-input-error>（エラー表示） ／ <x-auth-session-status>。 これらは認証機能の動作に必須。デザイン変更時に誤って削らないこと。

つまずき・気づき

Q. 緑/赤/グレーもtailwindに定義が必要？

A. 不要。Tailwind標準パレットに入っているのでそのまま使える。定義が必要なのはパレットに無いブランド色（オレンジ・ブルー・クリーム）と、便宜的なエンティティ名トークンだけ。

Q. const brand は extend の中に書く？

A. いいえ。export default の外（importの直後）に書く。オブジェクトの中（extend/colors）には変数宣言は書けない。

Q. 色を定義しただけで画面は変わる？

A. 変わらない。実際にクラス（bg-action 等）を使ってビルドして初めて反映される（未使用クラスはpurgeされる）。npm run dev で確認。

環境メモ：このプロジェクトは Tailwind v3 スタイル（tailwind.config.js ＋ @tailwind ディレクティブ）で動作。 package.json に @tailwindcss/vite ^4.0.0（v4用）と tailwindcss ^3.1.0 が混在しているが、実際はv3設定が効いている。今は触らないでよい。
