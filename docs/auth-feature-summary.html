★ 今回いちばん大事な教訓：「Breeze残骸」パターン

同じ根っこの失敗を何度も踏んだ。 Breezeが自動生成したコードには、こちらがカスタマイズで変更・削除した要素（name / email_verified_at / dashboard / rememberToken）が あちこちに残る。1か所直しても、同じ要素を参照する別の箇所が後から牙をむく。
変更した要素残骸が出た場所症状
name → usernameFillable / Controller / Blade / Factory / Test登録できない・列が無いエラー
email_verified_at 削除Factory / ProfileController / castsno such column エラー
dashboard → mypageRegisteredUserController ほかRoute not defined（500）
再発防止の鉄則：スキーマ名・ルート名・カラム名を変えたら、 必ず grep で全参照を洗い出してから変更する。 
grep -rn "古い名前" app routes tests resources/views database
「動いてる画面」だけ見て満足しない。その変更後に再テストしていない経路（登録後リダイレクト等）に残骸が潜む。

横断的に効く技術ポイント（復習）

HTML / フォーム

項目要点
type と name別物。type=入力の種類、name=送信フィールド名。混同すると平文表示等の事故
for と idラベルの for は入力の id と一致。id はページ内で重複禁止
autofocus最初にカーソルを置く欄を1つだけ
autocompleteログイン=current-password / 登録=new-password。メールログインのID欄は username
ログアウト状態変更なので POST（<a> 不可・@csrf 必須）

Laravel / Blade

項目要点
レイアウトコンポーネントguest（ログイン前）/ app（ログイン後）。枠を1か所直すと配下の全画面に反映
スロット{{ $slot }}（デフォルト）と <x-slot name="...">（名前付き）
名前付きエラーバッグパスワード=$errors->updatePassword->... / 退会=$errors->userDeletion->...
{{ __('...') }}翻訳ヘルパ＋出力。翻訳が無ければそのまま表示。v1は直接日本語でも可
Auth::check() / auth()->check()同じ動作。ファサードで統一すると解析しやすい
@props(['x'])デフォルト無し＝必須。渡されないと「未定義」エラー（:message単数の罠）

テスト

項目要点
Unit と FeatureUnit=クラス単体 / Feature=HTTP通し。認証チェックはFeature
テストDBSQLite メモリ。本番MySQLに触れない・使い捨て
RefreshDatabase各テストでDB初期化
論理削除の検証assertNull($user->fresh()) は不可。assertSoftDeleted() を使う
絞り込み実行--testsuite=Feature / --filter=メソッド名 / ファイル名指定
テストは仕様の写し鏡生成テストが落ちたら「実装バグ」か「テストが古い前提のまま」かを切り分ける

この後の開発で押さえるべきポイント

① レイヤー構成を守る（次の機能から本番）

本・知識・行動のCRUDからは、規約どおり ビジネスロジックを app/Services/ に分離する（Fat Controller禁止）。 Controller=受付、Service=ロジック、Model=データ、FormRequest=検証、Blade=表示。

② 機能単位の進め方を固定化する

Issue作成 → ブランチ（feature/(issue番号)-概要）→ 実装 → Featureテスト → PR（Closes #番号）。 認証で踏んだ「残骸」を避けるため、カラム/ルート名はgrepで全置換を習慣に。

③ エンティティ色とUI部品の再利用

本=緑 / 知識=青 / 行動=橙 を bg-book 等で一貫使用。 required-badge のように、繰り返す要素はコンポーネント化。入力欄スタイルも共通化を検討。

④ 残課題（忘れず対応）
  • My Pageの3コレクション行は href="#" の仮リンク。各CRUD実装時に route('books.index') 等へ接続
  • 未使用の dashboard 参照（VerifyEmail系・ConfirmablePassword・welcome.blade）が残存。潜在バグなので整理推奨
  • welcome.blade.php は未使用。削除してよい
  • Breezeが残した未使用コンポーネント（dropdown / nav-link 等）も整理可
⑤ アクセス制御・認可

未ログイン→ログイン画面リダイレクトは auth ミドルウェアで担保済み。 今後は「他ユーザーのデータにアクセスさせない」認可が必須（本・知識・行動は所有者のみ操作可）。 where('user_id', auth()->id()) やポリシーで制御する。

次の作業

認証基盤が固まったので、次は 本（Book）のCRUD から着手するのが自然。 一覧取得では N+1対策（with()）、ロジックは Service、検証は FormRequest、 そして各機能ごとに Featureテストを書く、という今回確立した型をそのまま展開していく。
