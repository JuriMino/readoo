1. 設計判断：どう並べ替えるか

方式の選択：項目ソート vs 手動ドラッグ

「並べ替え」には2つの解釈がある。着手前に整理して選んだ。

方式内容判断
項目でソートタイトル・登録日等の項目で昇順／降順。DB の ORDER BY で実現採用
手動ドラッグ並べ替え行をドラッグして任意順に並べ、表示順を保存見送り

理由： 読書ログは「著者で揃えて見たい」「読了日順に追いたい」という属性による整列が自然で、任意の手動順に意味が薄い。加えて手動順は display_order カラム追加・JS・ページネーションとの兼ね合いが必要で、実装コストが釣り合わない。

UI の選択：列ヘッダークリック vs ドロップダウン

3一覧はいずれもテーブル表示なので、列ヘッダー自体をクリック対象にするのが直感的と判断。矢印で現在のソート状態も同時に表現できる。

タイトル ▼ | 著者 ▼ | ステータス ▼ | 登録日 ▲
─────────────────────────────────
7つの習慣    | …     | 読了        | 2026-07-10

ドロップダウン方式（「並べ替え：登録日（新しい順）▾」）はカード形式の一覧やスマホ向きだが、今回はテーブルなので見送り。

2. 全体設計：3層でどう動くか

仕組みはシンプルで、URL のクエリパラメータを唯一の状態としている。セッションにも DB にも保存しない。

層役割
URL?sort=title&direction=asc&page=1 が「今どう並んでいるか」の唯一の情報源
コントローラークエリを受け取り、ホワイトリスト照合してから orderBy() に反映
ビュー共通コンポーネント <x-sort-header> が、次に押したときのリンクと矢印を描画
URL を状態にする利点： ソートした画面をそのままブックマーク・共有できる／ブラウザの戻るが自然に効く／ページネーションとの合流も withQueryString() だけで済む。サーバー側に状態を持たないので実装も単純になる。

3. 共通コンポーネント <x-sort-header>

3一覧すべてのヘッダーで使い回す中核。resources/views/components/sort-header.blade.php として新規作成した。

@props(['column', 'label', 'sort', 'direction'])
@php
    $isActive = $sort === $column;
    $nextDirection = ($isActive && $direction === 'asc') ? 'desc' : 'asc';
    $arrow = $isActive ? ($direction === 'asc' ? '▲' : '▼') : '▼';
@endphp
<th class="px-5 py-3 text-left font-bold whitespace-nowrap">
    <a href="{{ request()->fullUrlWithQuery(['sort' => $column, 'direction' => $nextDirection, 'page' => 1]) }}"
       class="inline-flex items-center gap-1 hover:text-gray-700 {{ $isActive ? 'text-gray-900' : '' }}">
        {{ $label }}
        <span class="text-xs {{ $isActive ? '' : 'text-gray-300' }}">{{ $arrow }}</span>
    </a>
</th>

各行の意味

要素解説
$isActiveこの列が「今ソートに使われている列」か。矢印の出し分けと文字色に使う
$nextDirection次にクリックしたときの向き。「アクティブかつ昇順」のときだけ降順、それ以外は昇順。つまり他の列から来たときは必ず昇順スタートになる
$arrow現在の状態表示。アクティブなら実際の向き（▲/▼）、非アクティブは薄いグレーの ▼
fullUrlWithQuery()現在の URL のクエリを維持したまま指定キーだけ上書き。将来「検索キーワード」等を足しても壊れない
'page' => 1並べ替えたら1ページ目に戻す。3ページ目でソートを変えて「3ページ目のまま」だと文脈を失うため
コンポーネント化した狙い： ヘッダー1つあたり「リンク生成・トグル判定・矢印描画」の3つの関心事がある。これを素の <th> にベタ書きすると、3一覧 × 十数列で同じロジックが散らばる。1ファイルに閉じ込めたことで、後の「折り返し禁止(whitespace-nowrap)を全ヘッダーに追加」が1行の修正で済んだのが実際に効いた。

4. 7/16 書籍一覧の実装

コントローラー：ホワイトリスト方式

public function index(Request $request)
{
    $this->authorize('viewAny', Book::class);

    // ホワイトリスト照合（インジェクション対策）
    $sortable = ['title','author','publisher','genre','status','started_at','created_at'];
    $sort = in_array($request->query('sort'), $sortable, true) ? $request->query('sort') : 'created_at';
    $direction = $request->query('direction') === 'asc' ? 'asc' : 'desc';

    /** @var \App\Models\User $user */
    $user = Auth::user();
    $books = $user->books()
        ->orderBy($sort, $direction)
        ->paginate(10)
        ->withQueryString();   // ページ送りでソート条件を維持

    return view('books.index', [
        'books' => $books,
        'sort' => $sort,
        'direction' => $direction,
    ]);
}
なぜホワイトリストが必須か（最重要）

Laravel のクエリビルダは値（バインドされる部分）は自動でエスケープするが、カラム名はしない。orderBy($request->query('sort')) のようにユーザー入力を直接渡すと、SQL の識別子として組み立てられSQL インジェクションの入口になる。「許可した名前と完全一致した場合だけ通す」形にすれば、入力が何であれ安全な値しか SQL に届かない。

もう一つの実利として、存在しないカラムを渡されて 500 になるのを防ぐ役割もある（実際この後、知識一覧でこれに救われず落ちた → つまづき④）。

withQueryString() の役割： これが無いと、ページネーションのリンクが ?page=2 だけになり2ページ目に進んだ瞬間ソートが解除される。付けると現在のクエリ（sort・direction）をページリンクに引き継いでくれる。

ソート対象の決定

当初案から、実際の使用感を踏まえて調整した。

列ソート備考
登録日 created_at対象デフォルトのソートキー（降順）。列自体も新規追加した
タイトル title対象

出版社 publisher対象途中で追加
ジャンル genre対象途中で追加
ステータス status対象

5. 7/16 付随した UI 調整

つまづき①：エディタに「9個のエラー」が出る

症状 赤い波線＋「Undefined method」「Expected type」が約9箇所

コントローラー修正後、エディタが大量にエラーを表示。しかし php -l は構文エラーなし、php artisan route:list も全ルート正常表示で、アプリは問題なく動いていた。

正体： VSCode の Intelephense による偽陽性。Intelephense は Eloquent の動的メソッド（$user->books() のようなリレーション、orderBy／paginate／withQueryString のチェーン）を静的に解決できず、片っ端から警告を出す。プロジェクトに PHPStan 等は未導入なので、赤線の出所はエディタで確定できた。

対応： Laravel 用のヒント定義を生成するツールを導入。

docker compose exec app composer require --dev barryvdh/laravel-ide-helper
docker compose exec app php artisan ide-helper:generate
docker compose exec app php artisan ide-helper:models --nowrite

…が、生成しただけでは波線が消えなかった。ファイル（_ide_helper.php 等）はホスト側に正しく出力されていたので、原因はIntelephense が新しいファイルを再インデックスしていないこと。

解決： Cmd + Shift + P → 「Developer: Reload Window」で再インデックスが走り、波線が消えた。

学び： 「エディタの赤線」と「本当のエラー」は別物。切り分けには php -l（構文）と artisan route:list 等（実際に起動するか）が有効で、動いているなら赤線は環境側の問題。また Intelephense はキャッシュが強く、ファイルを増やしたら Reload Windowまで含めて1セット。
後始末： _ide_helper.php は約1MBの自動生成物なので .gitignore に追加してコミット対象から除外した（_ide_helper.php / _ide_helper_models.php / .phpstorm.meta.php）。

つまづき②：max-w-8xl が存在しない

症状 max-w-7xl は手狭、max-w-8xl にしたら今度は広すぎる

実は Tailwind 標準に max-w-8xl というクラスは無い（標準スケールは 7xl＝80rem/1280px まで）。未定義クラスは単に何も適用されないので、実態は「幅制限なし＝全幅」になっていた。「広すぎる」と感じたのはそのため。

対応： tailwind.config.js に中間サイズとして自分で定義した。7xl（1280px）と全幅の間を取って 88rem（1408px）。

theme: {
    extend: {
        maxWidth: {
            '8xl': '88rem', // 1408px（7xl=1280pxと全幅の中間）
        },
    },
},

これで max-w-8xl が実際に効くようになった。

学び： Tailwind は未定義のクラス名を書いてもエラーにならず、黙って無視される。「効いていない」と「効きすぎている」の区別がつきにくいので、スケール外の値を使うときはconfig に定義するか任意値クラス（max-w-[88rem]）を使う。

つまづき③：日付が2行に折り返す

症状 2026-07-13 が 2026-07- / 13 と2行になる

ヘッダーの「読書開始日」「ステータス」等も同様に折り返していた。テーブルのセルは既定で折り返す設定で、ISO 形式の日付はハイフンが改行位置とみなされるため、列幅が足りないと割れる。

対応： 折り返し禁止 whitespace-nowrap を、日付セルと共通コンポーネントの <th> に付与。

<td class="px-5 py-4 text-gray-600 whitespace-nowrap">{{ $book->created_at->format('Y-m-d') }}</td>

副次効果として、日付セルを nowrap にするとその列の最小幅が日付1行分まで広がるため、ヘッダー側の折り返しも連動して解消した。空いた幅は可変幅のタイトル列が吸収する。

6. 7/17 知識・行動一覧の実装（JOIN 方式）

なぜ書籍一覧と同じやり方ではダメか

知識・行動の一覧には「参照元」列（＝その知識/行動の元になった書籍のタイトル）がある。これを並べ替えたいが、books.title は別テーブルの列なので orderBy('book') のようには書けない。knowledges テーブルに book という列は存在しないからだ。

そこで books テーブルを JOIN して、books.title で並べる方式に切り替えた。あわせて「ソートキー → 実カラム名」の対応表（$sortMap）を導入し、ホワイトリストの役割も兼ねさせた。

コントローラー（知識／行動 共通の形）

$sortMap = [
    'created_at' => 'actions.created_at',
    'title'      => 'actions.title',
    'book'       => 'books.title',   // 参照元 ← 別テーブルの列を指す
];

$sortKey = array_key_exists($request->query('sort'), $sortMap) ? $request->query('sort') : 'created_at';
$sortColumn = $sortMap[$sortKey];
$direction = $request->query('direction') === 'asc' ? 'asc' : 'desc';

$actions = Action::query()
    ->join('books', 'actions.book_id', '=', 'books.id')
    ->where('books.user_id', $user->id)
    ->whereNull('books.deleted_at')   // ★ 論理削除された書籍を除外
    ->with('book')
    ->select('actions.*')             // ★ 列名衝突を防ぐ
    ->orderBy($sortColumn, $direction)
    ->paginate(10)
    ->withQueryString();

return view('actions.index', [
    'actions' => $actions,
    'sort' => $sortKey,               // ビューにはキー('book'等)を渡す
    'direction' => $direction,
]);
$sortMap パターンの利点： 「URL に出す短いキー」と「SQL の実カラム名」を分離できる。URL には ?sort=book という人が読める短い名前を出しつつ、内部では books.title に解決する。array_key_exists で照合するので、マップに無いキーは通らない＝ホワイトリストとして機能する。ビューの <x-sort-header column="book"> ともキーで一致する。

★ 落とし穴1：JOIN するとソフトデリートの除外が消える

これが今回いちばん重要な発見

元のコードは whereHas('book', ...) で「自分の書籍に紐づくものだけ」を絞っていた。これは Eloquent 経由なので、Book モデルの SoftDeletes グローバルスコープが自動適用され、books.deleted_at IS NULL が暗黙に付いていた。

ところが join('books', ...) は生の SQL 結合で、モデルのスコープを一切通らない。そのまま置き換えると「論理削除した書籍に紐づく知識・行動が一覧に出てくる」という、エラーも出ない静かなバグになる。

対策： ->whereNull('books.deleted_at') を明示的に足して、失われたフィルタを手で補う。

学び： Eloquent の便利さ（グローバルスコープ・リレーション）はモデルを経由して初めて効く。パフォーマンスや JOIN 目的で生のクエリビルダに降りるときは、それまで暗黙に効いていた安全網が外れることを意識する。特に SoftDeletes は「消えるべきものが消えない」形で表面化するので、テストで気づきにくい。

★ 落とし穴2：JOIN による列名衝突

actions と books はどちらも id / title / created_at を持つ。JOIN すると両方の列が結果セットに混ざり、後勝ちで Eloquent モデルに詰められて行動の id や title が書籍のもので上書きされる（詳細リンクが壊れる等）。

対策： ->select('actions.*') で取得する列を自テーブルだけに限定する。JOIN は「絞り込みと並べ替えのため」に使い、取り出す値は汚さない。参照元の表示自体は ->with('book') の Eager Load 側が担当するので、これで両立する。

行動一覧での最適化

with(['book','knowledge']) → with('book')： 行動一覧のビューを確認したところ $action->knowledge を参照していなかったため、knowledge の Eager Load は不要と判断して外した。使わないリレーションの先読みは無駄なクエリ1本になる。（使っている場合に外すと N+1 になるので、必ず「ビューが参照しているか」を確認してから）

7. 7/17 つまづき集

つまづき④：Unknown column 'knowledges.book'

エラー SQLSTATE[42S22]: Column not found: 1054 Unknown column 'knowledges.book' in 'order clause'

$sortMap の記述ミス。参照元のマッピング先を、自テーブルのまま書いてしまっていた。

'book' => 'knowledges.book',  // ❌ knowledges に book 列は無い
'book' => 'books.title',      // ✅ JOIN した books のタイトル

JOIN・whereNull・select は正しく書けており、SQL のうち order by だけが誤りだったので、エラーメッセージの in 'order clause' から即座に切り分けられた。

学び： $sortMap の値は「SQL にそのまま出るカラム名」。キー（URL 用の名前）と値（実カラム）を混同しない。特に book のようなリレーション名に見えるキーは、つい 自テーブル.キー名 と書いてしまいがち。

つまづき⑤：何度クリックしても昇順にならない

症状 ▼ を押しても押しても降順のまま、▲ にならない 
$direction = $request->query('direction') === 'asc' ? 'desc' : 'desc';  // ❌
$direction = $request->query('direction') === 'asc' ? 'asc'  : 'desc';  // ✅

三項演算子の真の枝と偽の枝が両方 'desc' になっていた。URL は正しく direction=asc に変わっているのに、コントローラーが受け取った瞬間 'desc' に潰していた。

学び： 三項演算子の両枝が同じ値という誤りは、構文エラーにも実行時エラーにもならず、静かに「常に片方」になる。「操作しても何も変わらない」系の症状を見たら、分岐が本当に分岐しているかを疑う。切り分けには URL（＝入力）と表示結果（＝出力）のどちらがおかしいかを見るのが有効で、「URL は asc なのに降順」＝サーバー側、と絞り込める。

つまづき⑥：登録日が全部 2026-07-31 になる

症状 どの行も登録日が 2026-07-31（しかも未来の日付） 
{{ $knowledge->created_at->format('Y-m-t') }}  // ❌
{{ $knowledge->created_at->format('Y-m-d') }}  // ✅

PHP の日付書式 t は「その月の日数」を意味する。7月は31日あるので、7月に作られたレコードはすべて 2026-07-31 と表示されていた。日を出す書式は d。

書式意味7/17 の場合
d日（01〜31）17
tその月の日数（28〜31）31
学び： t は月末日の計算（Carbon::now()->format('t') で「今月は31日ある」）に使う書式。「全行が同じ値」かつ「その月の月末日」という症状は、この取り違えのサイン。データ側を疑う前に書式文字列を見る。

9. 既知の制約・今後の検討

① ジャンル・ステータスはコード値順

genre・status は DB にコード値（business、unread 等）で保存されているため、アルファベット順に並ぶ。画面表示の日本語ラベル順（未読 → 読書中 → 読了 のような意味のある順序）にはならない。まずは単純順で運用し、気になれば ORDER BY FIELD() 等で対応を検討する。

② 該当箇所（book_page）は今回ソート対象外

文字列カラム（varchar）なので、ソートすると文字列順（10 が 9 より前）になり直感に反する。今回は対象から外した。

③ 一覧ごとに幅が不揃い

Book は max-w-8xl（1408px）に広げたが、Knowledge・Action は max-w-7xl のまま。列を1つ増やしたので、揃えると余裕が出る。

④ 登録日セルの見た目が不揃い

Book は text-gray-600、Knowledge・Action は font-bold text-gray-900。日付がタイトル並みに目立っているので、統一するなら 2箇所を text-gray-600 に。

「実際に使ってみて不便だと感じた」ところから出発した機能なので、要件が具体的で判断に迷いが少なかった。実装面では、共通コンポーネントを最初に切り出したことが効き、3一覧への展開と後からの一括修正（whitespace-nowrap の追加）が安く済んだ。

一方で学びが多かったのはJOIN 方式に切り替えた知識・行動。whereHas から生の join に降りた瞬間、Eloquent が暗黙に効かせていた安全網（SoftDeletes のグローバルスコープ）が外れるという、エラーにならない類の落とし穴に当たった。「便利な抽象を下に降りるときは、抽象が何をしてくれていたかを棚卸しする」という原則を、身をもって確認できた。

つまづきの多くはエラーが出ないバグ（両枝が同じ三項演算子、t と d の取り違え、未定義の Tailwind クラス）だった。共通するのは「症状の形から原因を逆算できる」こと。全行が月末日なら書式、押しても変わらないなら分岐、効いていないなら定義の有無。次回以降の切り分けに活かしたい。
