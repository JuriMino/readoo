1. 一覧ビュー（index）

knowledges/index.blade.php を土台に、テーマ色をオレンジへ差し替えて自作。列は Figma に合わせて タイトル / 参照元 / 該当箇所 / タグ / 詳細 の5列。

重要ポイント：一覧では ?-> が不要な理由

詳細（show）では $action->book?->title のようにヌル安全が必須だが、一覧では $action->book->title で良い。理由はController側で whereHas('book', ...) を掛けており、論理削除された本の行動はそもそも取得されないから。各行の book は必ず存在する。「どこで ?-> が要る／要らないか」は "そのクエリで null になり得るか" で判断する。

設計メモ： thead の背景を bg-gray-50 ではなく bg-orange-50 text-orange-800 にしてテーマ色を反映（Figma準拠）。タグpillも青→オレンジ（border-orange-200 bg-orange-50 text-orange-600）。「知識一覧の複製」ではなく「同じ骨格・別テーマ」という意識で差分を作る。

2. 新規登録ビュー（create）＝本日の山場

Figmaの構成そのままに、参照元 / 行動の内容 / 5W1H / タグ の4カード。核は 本を選ぶと知識プルダウンが絞り込まれる連動ドロップダウン（Alpine.js）。

5W1H → カラム対応

画面ラベルカラム必須/任意
WHEN：いつ実行する？timing必須
WHERE：どこで実行する？place必須
WHO：誰に関わる？target_person任意
WHAT：何をする？detail任意
WHY：なぜやる？reason必須
HOW：どうやってやる？method必須

この対応は ActionRequest の rules と完全一致していることを事前に確認してから実装した。

Alpine連動の仕組み（3つの部品）

① データを渡す

サーバから自分の全知識を id / book_id / title の3列だけ取得し、@js($knowledges) でJSON化してAlpineへ。@js はXSS安全にJS値へ変換してくれるBladeディレクティブ。

② 絞り込む（getter）

get filterKnowledges(){
    return this.knowledges.filter(k => String(k.book_id) === String(this.selectedBook));
}

選択中の本に属する知識だけを返す。String(...) === String(...) は型の吸収：old()由来は文字列、@jsのbook_idは数値なので両方を文字列化して比較する（Blade側の == と同じ発想をJSでやっているだけ）。

③ 描画する（x-for）

<template x-for="k in filterKnowledges" :key="k.id">
    <option :value="k.id" x-text="k.title"></option>
</template>

selectedBook が変わると getter が再計算され、プルダウンが描き直される。

本を変えたら知識をリセット： 本セレクトに x-on:change="selectedKnowledge = ''" を付与。本を切り替えたのに前の本の知識が残ると不整合になるため。初回表示（知識詳細から来て両方プリセット）では change は発火しないので、プリセットは壊れない。

3. 詰まったところ：画面は出るのに動かない タイプミス4種

写経後「画面表示OK」で油断したが、送信と連動が丸ごと死んでいた。表示だけでは絶対に気づけない4つ。今後の教訓として最重要。

#誤正症状
1 <form action=""> action="{{ route('actions.store') }}" 送信先が現在URL(/actions/create)になりPOSTを受けるルートが無く405エラー
2 x-modal="..." ×2 x-model="..." 存在しない指定。selectedBookが更新されず絞り込みが一切効かない
3 value="k.id" :value="k.id" コロン無しだと全optionのvalueが文字列 "k.id" に固定。知識IDが入らずバリデーションで弾かれる
4 bg-gray-200(枠なし) bg-gray-50 border border-gray-200 知識セレクトだけ見た目が浮く（機能は動く・軽微）
今日いちばんの教訓：「表示OK ≠ 完成」

Bladeは壊れた属性（x-modal、action=""、value="k.id"）でもそのままHTMLとして描画してしまう。エラーも警告も出ない。だから 送信できるか と 連動が動くか は、目視ではなく必ず手で操作して確認する。特にAlpine（x-model/:value）とフォーム送信先（action）は、写経で最も事故りやすい3点セット。

コロン(:)の意味（再確認）： Alpineで :value="k.id" の : は x-bind:value の略。「中身をJS式として評価してバインドする」印。付け忘れると「ただの文字列」になる。:key・:value・:class など、動的な値には必ずコロン。

4. 設計判断：createで本を「固定」するか「選択可」か

Q（自分）： 知識登録では本詳細から来たら本を固定したはず。行動でも本詳細→本固定、知識詳細→本＆知識固定にする方が一貫するのでは？

事実確認： knowledges/create は実は「固定（ロック）」しておらず、初期選択されるだけの通常セレクト（変更可）だった。「変更不可の表示」にしていたのは知識の edit 画面。

結論： create は選択可のままでよい。理由は—

  • edit＝すでに確定した参照。後から付け替えると履歴の意味が崩れる → ロックが正しい
  • create＝まだ何も確定していない下書き。初期選択は「入り口からの気の利いたプリセット」にすぎず、選び直せて当然 → 変更可でよい

これで knowledges/create とも思想が一致し、ロック実装（disabled＋hidden）による分岐の複雑化も回避できた。

5. 3つの入り口ごとの初期挙動

入り口from初期状態
独立一覧からindex本・知識とも未選択。本を選ぶと知識が絞られる
本詳細からbook本が初期選択。知識はその本の分に絞られた状態
知識詳細からknowledge本・知識とも初期選択済み

いずれも「初期選択」であって「ロック」ではない。from は登録後・キャンセル時の戻り先の分岐にも使う（$cancelUrl の match）。

サーバが最終防衛線： Alpineの絞り込みは利便性の層にすぎない。フォームを改ざんして他人の本の知識IDを送っても、ActionRequest の Rule::exists('knowledges','id')->where('book_id', ...) が弾く。JSは信用せず、正しさはサーバで担保する。
