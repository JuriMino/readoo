2. 設計検討：本の論理削除で子も論理削除するか

結論：v1は現状維持（フックなし＋?->）

実現は可能（Book に booted() の deleting フックを10行ほど）。だが「削除のカスケードは安いが、復元のカスケードが高い」のが本質。

論点中身
① 復元の非対称性削除で子も消すなら復元で子も戻したくなるが、単純復元は「本を消す前から個別削除していた子」まで過剰復元してしまう（古典的な罠）。restore 未実装の今は顕在化しない
② 物理削除の切り分け退会の forceDelete 時も deleting は発火。isForceDeleting() でガードしないと二重処理
③ 知識→行動の並行問題「本→子」を入れるなら「知識→行動」も揃える必要が出る

現状は whereHas('book') の除外 ＋ Policy の ?-> 403 で、ユーザーからは既に「本を消すと子も消えたように見える」。見た目のゴールは達成済みなので、データ上の孤児は退会時の物理削除に任せる。

3. ActionController 深掘り Q&A

Q. index() の whereHas / with は何をしている？

whereHas＝絞る / with＝読む、の2つは別の仕事
  • whereHas('book', 自分)：SQLの EXISTS サブクエリを生成する絞り込み。「自分の本の行動だけ」に限定。本のデータは取ってこない（判定に使うだけ）
  • with(['book','knowledge'])：eager load（先読み）で N+1 回避。表示用データをまとめて取得
  • だから book は両方に登場する（絞るのと読むのは別だから）
  • with('knowledge') は絞り込まない。knowledge_id が null の行動も出て、その行の $action->knowledge が null になるだけ
  • whereHas の EXISTS には books.deleted_at IS NULL が自動で入る → 本が論理削除された行動は一覧に出ない（昨日の孤児の話と接続）

Q. create() の authorize('viewAny') は、プルダウンの一覧データが必要だから？

いいえ。認可（authorize）とデータ取得（->get()）は別物
  • authorize() は Policy を呼んで true/false を判定するだけ。何も取得しない
  • viewAny を使う理由は「フォーム表示時点では認可対象が無いから」。view/update/delete は具体的な $action が要る、create(User,Book) は具体的な $book が要るが、本はフォームの中で選ぶため表示時点では未確定
  • → 粗いゲート viewAny（return true）で通す
  • セキュリティは3段構え：authミドルウェア／自分のデータしか出さないプルダウン／本が確定する store() での authorize('create', [Action::class, $book])
  • 原則：認可のタイミングは「対象が確定する場所」に置く

Q. $knowledges のクエリは何をどう取っている？

間接所有でたどり、Alpine連動用に最小3カラムで取る 
Knowledge::whereHas('book', fn($q) => $q->where('user_id', $user->id))
    ->get(['id', 'book_id', 'title']);
  • 知識に user_id は無いので 知識→本→持ち主とたどって「自分の知識」を特定（間接所有）
  • 取得は自分の全知識（1冊に絞らない）。本ごとの絞り込みは後で Alpine（クライアント）が book_id で行う
  • 3カラムの用途：id＝option の value(=knowledge_id) ／ book_id＝Alpine の絞り込みキー ／ title＝表示ラベル
  • 最小3カラムなのは、JSONでクライアントに渡すため。content やタグまで取ると重い＋不要な露出
  • 役割分担：サーバは素材を最小限で渡す／絞り込みの見せ方はクライアント（＝「Alpine＝便利さ／バリデーション＝正しさ」と一貫）

Q. 知識詳細から来て knowledge_id が固定なら、book_id も必然的に固定？

はい。知識は必ず1冊の本に属す（book_id は NOT NULL）から
  • 知識が1つ決まる → その book_id は一意 → 参照元の本も自動的に確定（本は知識に従属）
  • 3入口のうち知識詳細だけ「知識→本」の向きで本が決まる。本の詳細／独立一覧は「本→知識」の向き
  • 整合ルール Rule::exists('knowledges','id')->where('book_id', ...) により、知識詳細から来たセットは必ず通り、book_id すり替えは弾かれる

Q. store() の $request->validated() は？ safe() は何をする？

validated()＝rules を通った項目だけの検証済みデータ。safe()＝それを絞れる形で返す
  • validated() はリクエスト全体ではなく、rules() にキーがある項目だけを返す。だから book_id/knowledge_id は入るが、rules に無い from は入らない（＝store では from を $request->input('from') で読む理由）
  • store に入った時点で検証は通過済み。失敗時は store が動く前に自動で差し戻される
  • safe() は validated と同じデータを、->except()/->only() で加工できるオブジェクトで返す（validated() はただの配列）
  • $request->safe()->except(['book_id','knowledge_id']) ＝ 検証済みから2つを除いたFillable な中身カラムだけを create に渡す
  • book_id はリレーション経由で、knowledge_id は後で個別代入するので除外。all() を使わず safe() 起点にするのがマスアサインメント防御
