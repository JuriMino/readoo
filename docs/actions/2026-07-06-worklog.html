学びポイント（今日の中心）

① Factory の state とは「既定値のバリエーション（名前付きの上書き）」

definition() が基本形。state はそこに「一部だけ差し替えたパターン」を名前を付けて生やしたもの。$this->state(クロージャ) が返す配列を definition の上にマージする（書かない項目は definition のまま）。

public function forKnowledge(Knowledge $knowledge): static
{
    return $this->state(fn () => [
        'book_id'      => $knowledge->book_id,   // ここだけ
        'knowledge_id' => $knowledge->id,        // 上書き
    ]);
}

// 使い方（チェーンで呼ぶ）
Action::factory()->forKnowledge($k)->count(3)->create();

Laravel 標準の ->trashed() や ->count() も同じ仲間。「factory に生やす名前付きバリエーション」とイメージする。今回の forKnowledge は「book_id を知識の本に合わせる」ので、knowledge_id => Knowledge::factory() のような本の食い違いを防げる。

② メソッドの返り値の型指定（: void / : static など）

メソッドの () の後ろの : 型 は「何を返すか」の宣言。

型意味例
: void何も返さない（値を return しない）up() / テストメソッド
: array配列を返すdefinition() / rules()
: static自分と同じクラスを返す（＝チェーンできる）forKnowledge()
: bool真偽値を返すPolicy / authorize()
: BelongsTo / : HasManyそのリレーションオブジェクトを返すbook() / actions()

static は「呼び出し時の実際のクラス」を返す（継承先でも正しい）。似た self は「書いた場所のクラスに固定」。チェーンする / 継承される可能性があるものは static が定石。型を書く利点は「自己ドキュメント・宣言と違う型を返すと即エラー・IDE補完」。

③ コードを書く前の「設計判断」— 案A→案Bへ切り替えた過程

行動は book_id 必須・knowledge_id 任意。「どこから作り始めるか（入口）」で設計が変わるため、routes を書く前に入口3種を洗い出して仕様を固めた。

当初は「独立一覧からは knowledge_id を null 固定」（案A・JS不要）を推したが、Figma を見ると 関連知識 の説明文に「本を選択後に絞り込まれます」とあり、連動ドロップダウン（案B）が前提だった。案Aだと「独立一覧から来たときだけ関連知識が死ぬ」＝画面の一貫性が崩れると判明。

案Bに切り替え。懸念だった「JSコスト」は このアプリが既に Alpine.js を使っている（削除モーダルの x-data 等）ため新依存なしで解決できると分かった。

役割分担の原則：UXとバリデーションを分ける
  • Alpine（クライアント）＝「便利さ」担当：本を選ぶと知識を絞り込む見た目
  • サーバのバリデーション＝「正しさ」担当：クライアントは信用しない前提で、送信値を必ず検証

→ 入口が3種あっても、サーバ側の検証は1本に統一できる。「フォームの見せ方（固定 or 選択）」の違いにすぎない、と整理できたのが設計上の収まりどころ。

1. 作成フローの確定仕様

入口frombook_idknowledge_id戻り先
知識詳細からknowledgeその知識の本に固定その知識に固定knowledges.show
本の詳細からbookその本に固定その本の知識から選択（null可）books.show
独立一覧からindex本を選択（必須）選んだ本の知識から選択（null可・本未選択なら選べない）actions.index
絶対に守る不変条件

knowledge_id を入れるなら、その知識は book_id の本のものでなければならない（knowledge.book_id === action.book_id）。これが崩れると「Aの本の行動なのにBの本の知識に紐づく」矛盾データになる。3入口すべてでこの整合を担保するのが設計の肝。

2. routes/web.php

Knowledge と同じ flat な resource。入口3種はフォームの from / book_id / knowledge_id で吸収するため、ネストは不要。

use App\Http\Controllers\ActionController;   // use 群に追加

// auth グループ内
Route::resource('actions', ActionController::class);

3. ActionRequest（整合バリデーションの心臓部）

POST（新規登録）の時だけ、参照元の本と（任意の）知識を検証する。

if ($this->isMethod('post')) {
    // 本：必須かつ「自分の本」に限定
    $rules['book_id'] = [
        'required',
        Rule::exists('books', 'id')->where('user_id', $this->user()->id),
    ];
    // 知識：任意。送られてきたら「その book_id の本に属する知識」に限定
    $rules['knowledge_id'] = [
        'nullable',
        Rule::exists('knowledges', 'id')->where('book_id', $this->input('book_id')),
    ];
}
  • book_id → 「自分の本か」を検証
  • knowledge_id → 「送られた book_id の本に属する知識か」を検証
  • 合わせると「自分の本 ＋ その本の知識」しか通らない。Alpine をすり抜けても矛盾データを弾ける
5W1H とカラムの対応

WHEN=timing / WHERE=place / WHO=target_person / WHAT=detail / WHY=reason / HOW=method。必須は WHEN・WHERE・WHY・HOW（NOT NULL）、任意は WHO・WHAT（nullable）。

4. 写経バグの修正

【factory】target_persom → target_person（typo）

Factory は内部で unguarded で動くため、typo したキーもそのまま属性化 → INSERT に実在しないカラムが混入し Unknown column 'target_persom' でSQLエラー。make() は通っても create() で落ちる。

【Request】早期 return で以降がデッドコード化

$rules = [...] とすべきところを return [...] にしてしまい、その時点で関数が終了。book_id / knowledge_id の検証ブロックが一度も実行されない状態になっていた。

実害：book_id が検証されず store で $validated['book_id'] が存在しないキーになる／整合チェックが丸ごと消えてセキュリティ穴。修正＝「配列は $rules にしまう → POST時だけ追記 → 最後に return」。「先頭で return すると後続が死ぬ」という基本挙動がハマったケース。
