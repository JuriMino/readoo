1. 詳細ビューの構成

知識詳細（knowledges/show）を土台に、テーマをオレンジへ。ただし行動詳細には独自ポイントが2つある。

独自ポイント① 5W1H は「小ボックス6個の2列グリッド」

登録画面の入力欄をそのまま読み取り専用にした見た目。6項目をベタ書きせず、配列＋ループで生成した。

@php
    $w5h1 = [
        'timing'        => 'WHEN：いつ実行する？',
        'place'         => 'WHERE：どこで実行する？',
        'target_person' => 'WHO：誰に関わる？',
        'detail'        => 'WHAT：何をする？',
        'reason'        => 'WHY：なぜやる？なんのためにやる？',
        'method'        => 'HOW：どうやってやる？',
    ];
@endphp
@foreach ($w5h1 as $name => $label)
    ...
    {{ $action->{$name} ?? '-' }}
@endforeach
動的プロパティ $action->{$name}： カラム名を変数で持ち、{$name} で動的にプロパティ参照。6個の似た項目を1つのループにまとめられる。空（WHO/WHATが未入力）のときは ?? '-' で「-」を表示、whitespace-pre-wrap で改行を保持。
?? が効く理由： WHO/WHAT は nullable。フォームで空欄のまま送ると Laravel 既定の ConvertEmptyStringsToNull ミドルウェアが空文字→null に変換して保存するため、?? '-'（null合体）でちゃんと「-」になる。もし空文字 '' のまま保存される設計なら ?:（falsy判定）が必要になるが、本アプリは前者なので ?? でOK。

独自ポイント② 「紐づく知識」だけ青テーマ・条件表示

@if ($action->knowledge)
    <section ...>
        <h2 class="... bg-knowledge text-white ...">紐づく知識</h2>
        <div class="... bg-blue-50">
            <a href="{{ route('knowledges.show', $action->knowledge) }}" ...>
                {{ $action->knowledge->title }}
            </a>
        </div>
    </section>
@endif
  • 色で「別ドメインへの導線」を表現： 他のカードはオレンジ（action色）だが、この1枚だけ bg-knowledge（青）。知識詳細へ飛ぶリンクであることを視覚的に区別。
  • @if ($action->knowledge) で条件表示： 関連知識は任意（knowledge_id は nullable）。紐づかない行動もあるので、ある時だけカードを出す。

#誤正症状
1 $action->knowkedge->title $action->knowledge->title knowkedge（l抜け＋k余分）は存在しないプロパティ→null→->titleで500エラー。@ifを通った直後なので知識が紐づく行動を開くと必ず落ちる

他に軽微な表記ゆれ（gap-yp-5→gap-y-5、パンくず Deatil→Detail、削除確認文の鉤括弧の位置）も修正。

3. ?->（ヌル安全）の要否・再整理

show の $action->book->title は ?-> 無しでも動く。理由は、本が論理削除されている場合は Policy の view が先に403を返し、この画面に到達しないから（$user->id === $action->book?->user_id が false になる）。よって描画時点で book は必ず存在する。

整理すると：

場所?->理由
Policy（view/update/delete）必須ここで null 判定して403を出す“関所”。?->が無いと500になる
一覧（index）不要whereHas('book') で論理削除済みの本は取得されない
詳細（show）不要（付けても可）Policyの関所を通過済みなので book は存在する。防御的に付けてもよい

「そのコードに到達した時点で null になり得るか」で判断する、という原則。

4. 次の edit ビューの設計方針（確認済み）

方針： edit は「すでに確定した参照」を扱うので、create とは扱いを変える。
  • 本・関連知識はロック（読み取り専用表示） — 確定済みの参照を後から付け替えない（履歴の意味が崩れるため）。知識の edit と同じ思想。
  • タイトル・該当箇所・5W1H・タグは編集可
  • @method('patch')、old('field', $action->field) で現在値を初期表示
