1. よく使うテストメソッド全解説

1本のテストは「①前提を作る（Arrange）→ ②操作する（Act）→ ③確かめる（Assert）」の3段。使ったメソッドを役割ごとに整理する。

① 前提・操作系（リクエストを組み立てて送る）

メソッド役割
actingAs($user)以降のリクエストを「$user でログイン済み」として送る。認証をくぐらせる。
from($url)「直前にいたURL（リファラ）」を偽装する。バリデーション失敗などで「元の画面へ戻る」挙動の戻り先を決めるために置く。
get / post / patch / delete($url, $data)指定メソッドでフルスタックを通してリクエスト。第2引数はフォーム送信データ。返り値は TestResponse。
2つの "from" を混同しない：
  • ->from($url)＝テストのヘルパー。バリデーション失敗時の戻り先（リファラ）を偽装するだけ。
  • ペイロードの 'from' => 'index'＝アプリ独自のhiddenフィールド。登録成功後の戻り先を分岐させる。
名前は同じでも全くの別物。バリデーションが失敗するテストではコントローラ本体は実行されないので、後者（アプリのfrom分岐）は動かず、戻り先は前者（->from()）で決まる。

② レスポンス検証系（TestResponseに対して）

メソッド合格条件
assertOk()HTTPステータス 200（正常表示）
assertSee($text) / assertDontSee($text)レスポンス本文にその文字列が含まれる／含まれない。一覧の「自分のは出る／他人のは出ない」検証に使用
assertRedirect($url)指定URLへリダイレクト（3xx）している。$urlは必ず route('...') で作る（生文字列だとドットもそのまま比較され失敗）
assertForbidden()HTTPステータス 403（認可拒否）。Policyで弾かれたことの確認
assertSessionHasErrors($keys)セッションに指定キーのバリデーションエラーがある。['title','timing',...] と配列で複数同時にも確認できる

③ DB検証系（TestCaseに対して＝$this->）

メソッド合格条件
assertDatabaseHas($table, $attrs)その属性に一致する行が存在する。登録・更新が「実際にDBに入ったか」を確認（['book_id' => null] のようにNULL一致も可）
assertDatabaseCount($table, $n)そのテーブルの行数がちょうど $n。('actions', 0) の 0 は「レコード件数」＝「1件も作られていない」の確認
assertSoftDeleted($model) / assertNotSoftDeleted($model)deleted_at が入った／入っていない。論理削除の成否を確認
兄弟メソッド： 今回は使っていないが assertDatabaseMissing($table, $attrs)（一致行が無いことの確認）もよく使う。Has／Missing／Count は3点セットで覚えると良い。

2. 例：整合性テストの逐次読み

$this->actingAs($user)                          // ① $user としてログイン
    ->from(route('actions.create'))            // ① 失敗時の戻り先＝create をセット
    ->post(route('actions.store'), $this->validData($book1, [  // ② 有効データ＋…
        'knowledge_id' => $knowledgeOnBook2->id,             //    別の本の知識を混ぜて送信
    ]))
    ->assertSessionHasErrors('knowledge_id')   // ③ knowledge_id でエラーが出た
    ->assertRedirect(route('actions.create')); // ③ create へ差し戻された

$this->assertDatabaseCount('actions', 0);      // ③ 行は1件も作られていない
何を保証しているか： Alpineの絞り込み（＝画面上の親切）を無視してフォームを改ざんし「book1の行動にbook2の知識」を送っても、ActionRequest の Rule::exists('knowledges','id')->where('book_id', ...) がサーバ側で弾く。「他人の本の知識」も「自分の別の本の知識」も、この1本のルールで同時に防げていることを、DB件数0で裏取りしている。

3. Tips：テスト観点の洗い出しと Unit / Feature の選び方

なぜ今回 Unit ではなく Feature なのか

UnitテストFeatureテスト
対象1つのクラス／メソッドを単体でルート→ミドルウェア→FormRequest→コントローラ→Policy→Eloquent→DB→レスポンスの通し
速さ速い（DB・HTTP無し）やや重い（フレームワーク起動・DB使用）
得意分岐の多い純粋なロジック（計算・状態遷移・値オブジェクト）部品の連携で意味が生まれる振る舞い（認可・検証・リダイレクト・永続化）

結論： Readooのこの機能の価値は「認可されているか」「不正入力を弾くか」「正しくDBに入る／戻るか」という統合の正しさにある。これらは部品を組み上げて初めて意味を持つので、Featureが適所。逆に、たとえばPolicyの $user->id === $action->book?->user_id だけを単体テストしても、「そもそもルートが認証で守られているか」「Fillableから漏れていないか」といった本当のリスクは捕まえられない。

Unitが向く場面（今後）： 「読書ストリークの日数計算」「ステータス遷移の可否判定」のような、DBやHTTPと切り離せるリッチなドメインロジックを作ったとき。現状のReadooはCRUD中心でその手のロジックが薄いので、Featureに寄せるのが費用対効果として妥当。

Featureテストはユースケース起点で洗い出す（＋「不変条件」起点）

質問「ユースケースベースに洗い出すのか？」への答え：基本はYes。ただし “叶えたい約束” と “絶対に起きてはいけないこと” の両面で拾う。

観点の型（この順で機械的に拾うと漏れにくい）：
  1. 正常系（ハッピーパス）：各操作が成功し、結果が残る。
    例：登録できる／更新できる／論理削除できる／一覧・詳細が表示できる
  2. 境界・検証（バリデーション）：必須が欠けたら弾かれ、DBに残らない。
    例：title・5W1H必須が空 → エラー＋件数0
  3. 認可（誰がやるか）：所有者以外は403、未ログインはログインへ。
    例：他人の行動を view/edit/update/delete → 403
  4. 業務ルール・整合性（そのドメイン固有の“不変条件”）：
    例：別の本の知識は紐付け不可／更新で本・知識は付け替え不可／入り口(from)で戻り先が変わる
  5. 否定の永続化確認：拒否された操作のあとにDBが変わっていない。
    例：403のあと assertDatabaseHas(元の値)／assertNotSoftDeleted／Count 0
効率よく絞るコツ：
  • 同値分割：全フィールドを個別に試さない。「必須が欠けた」代表1本（複数キーをまとめて assertSessionHasErrors([...])）で十分。
  • リスクベース：バグの起こりやすさ×被害の大きさが高いところに厚く。認可とマスアサインメント（Fillable漏れ）は被害が大きいので必ず書く。
  • 1テスト1論点：名前が「何を保証するか」を語れる粒度に割る（test_user_cannot_attach_knowledge_of_another_book のように）。
  • 成功と“変化なし”はペア：「できる」を書いたら、対になる「できない＆DB不変」も書くと守りが完成する。

4. 写経で出たエラーと修正

症状原因修正
405 Method Not Allowed（更新テスト）更新ルートは PATCH|PUT 専用なのに ->post() で送っていた->patch() に変更
リダイレクト先の文字列不一致（整合性テスト）assertRedirect('actions.create') と生文字列を渡し、/actions.create と比較していたassertRedirect(route('actions.create')) に変更
学び： 405は「URLはあるがメソッド違い」。同じ /actions/{action} でも show=GET / update=PATCH,PUT / destroy=DELETE とメソッドで振り分く。テストは ->patch() 等で狙ったメソッドを直接叩ける（ブラウザの @method('PATCH') 偽装に相当）。URL比較は必ず route() で作る。
