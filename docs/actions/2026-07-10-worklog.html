1. 編集ビュー（edit）

Figmaでは本・知識がプルダウンで描かれていたが、v1は変更不可（読み取り専用表示）で実装。知識のeditと同じ思想。

なぜ create は選択可で edit はロックなのか（再確認）： create＝まだ確定していない下書きなので選び直せて当然。edit＝すでに確定した参照で、後から付け替えると履歴の意味が崩れる。だから edit だけロックする。

実装の要点

  • 本・知識は <select> ではなく表示専用 <div>（bg-gray-100 text-gray-500）。name属性が無い＝送信されないので、コントローラの update() の ->except(['book_id','knowledge_id']) と合わせて二重に安全。
  • old($name, $action->{$name}) … 第2引数が「初期値＝現在のDB値」。エラー差し戻し時は入力値を優先、通常時は現在値を表示。5W1Hは動的プロパティなので波括弧 {$name} が要る。
  • method="post" ＋ @method('PATCH') … HTMLはPATCHを直接送れないのでPOST＋メソッド偽装。ActionRequest の book_id/knowledge_id ルールは if($this->isMethod('post')) の中なので、PATCH時は本・知識の必須チェックが走らない（＝送らなくてOK）。全部が整合している。
  • 関連知識が null の行動は $action->knowledge?->title ?? '(紐付けなし)' で表示。
写経の詰まり：Tailwindクラスの静かなタイプミス。 機能バグはゼロだったが、クラス名の綴りミスが6か所（gary／test／ronded／space-yp）。エラーも警告も出ず「ただ効かないだけ」で、枠線が消える・角丸が効かない・文字色が付かない等の崩れになる。text→test、gray→gary、rounded→ronded は特に出やすい。表示崩れを見たらまずクラス名の綴りを疑う。

2. 詳細への「紐づく行動」セクション

本詳細・知識詳細に、その本／知識から生まれた行動の一覧＋+ 行動を追加ボタンを設置。既存の「紐づく知識」セクション（<ul>リスト）と同じ形式に揃えてページ内の一貫性を優先（Figmaはテーブルだが、既に承認済みのリスト形式に合わせた）。

入り口パラメータが肝：
  • 本詳細から → from=book ＋ book_id … create で本が初期選択、知識はその本の分に絞られる
  • 知識詳細から → from=knowledge ＋ book_id ＋ knowledge_id … 本も知識も初期選択
これで前回作った create の「3つの入り口」のうち book / knowledge の2つが実データで検証可能になった。

eager load（N+1防止）

// BookController@show
$book->load([
    'knowledges' => fn($query) => $query->latest(),
    'actions'    => fn($query) => $query->latest(),
]);

// KnowledgeController@show
$knowledge->load(['actions' => fn($query) => $query->latest()]);
$knowledge->book_id はヌル安全不要： 知識は book_id カラムを必ず持つ（NOT NULL）ので、+ 行動を追加 のリンクにそのまま使える。$x->book?->... の ?-> が要るのは「リレーションを辿った結果がnullになり得る」場合（＝本が論理削除されているケース）で、カラム値そのものの話とは別。カラム値の有無とリレーション取得結果のnullは別物、という前回の整理どおり。

3. 全リンクの結線（href="#" 撲滅）

grep -rn 'href="#"' resources/views/ で残0件を確認。

箇所リンク先
knowledges/index・books/index 右上「Action List」route('actions.index')
actions/index の + New Action・空状態route('actions.create')
actions/index 各行「詳細」route('actions.show', $action)
補足： actions/index の New Action は ['from' => 'index'] を明示していないが、create 側の hidden が value="{{ request('from', 'index') }}" でデフォルトが index のため挙動は正しい。明示すれば「3つの入り口が対称に読める」可読性メリットはあるが機能差はゼロ。
