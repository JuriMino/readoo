① モデル別テーマ背景の導入

書籍・知識・行動それぞれで一覧〜詳細画面の背景色を変える方針になったため、 ページごとに背景クラスを書くのではなく、レイアウト側で一元管理する形にした。

// app/View/Components/AppLayout.php
class AppLayout extends Component
{
    public function __construct(
        public string $theme = 'base',
    ) {}
    ...
}
{{-- layouts/app.blade.php --}}
@php
    $bodyBg = match ($theme) {
        'book'      => 'bg-book/10',
        'knowledge' => 'bg-knowledge/10',
        'action'    => 'bg-action/10',
        default     => 'bg-base',
    };
@endphp
<body class="min-h-screen flex flex-col {{ $bodyBg }} font-sans text-gray-900 antialiased">
色は固定値でなく既存のbrandカラーを流用： tailwind.config.js に定義済みの book/knowledge/action（brandカラー）に 透過度修飾子（bg-book/10）を掛けるだけで薄色背景が作れる。 Tailwind v3はカスタムhexカラーにも /10 のような透過度指定が効くため、 bg-green-50 のような別カラーを新規定義せずに済み、知識・行動側もキーを変えるだけで展開できる。

各画面は <x-app-layout theme="book"> のように指定するだけでよく、 profile/edit.blade.php など theme 未指定の画面は従来どおり bg-base（クリーム色）のまま。

x-sort-header への color prop 追加

このコンポーネントは知識・行動一覧とも共有しているため、色をハードコードせず color propを追加してデフォルト gray（既存の見た目を維持）とし、 書籍側だけ color="book" を指定する形にした。

@props(['column', 'label', 'sort', 'direction', 'color' => 'gray'])
@php
    $colorClasses = [
        'gray' => ['active' => 'text-gray-900', 'hover' => 'hover:text-gray-700', 'arrow' => 'text-gray-300'],
        'book' => ['active' => 'text-book',      'hover' => 'hover:text-book',      'arrow' => 'text-green-300'],
    ];
    $c = $colorClasses[$color] ?? $colorClasses['gray'];
@endphp
ページネーションの表示件数バグ： 当初 array_rand 的な単純ループでウィンドウ計算をミスすると全ページが表示されてしまうため、 $current を中心に前後2ページ（最大5個）だけ表示するようclampして解消。

日付inputの年桁数バグ（Chromeの既知挙動）

読書開始日・読了日を手入力すると「202607/02/02」のように壊れる不具合があった。 原因は <input type="date"> に max 属性が無いと、 Chromeが年の最大値を仕様上の上限（西暦275760年）とみなし、年フィールドが6桁まで入力を受け付けてしまうため 4桁で自動的に月へ移動しない、というブラウザ側の仕様だった。

直すのは max であって min ではない。 minは下限のみを制御し、年の桁数上限には影響しない。 未来日も許容したかったため、実質無制限に近い max="9999-12-31"（4桁の範囲に収まる最大値）を指定し、 桁数バグの解消と未来日許容を両立した。 
<input type="date" name="started_at" ... min="1900-01-01" max="9999-12-31" ...>

④ 書籍詳細（show.blade.php）

紐づく知識／行動：リスト形式 → テーブル形式

Figmaでは「タイトル／該当箇所／タグ／詳細」の列を持つテーブルで、 ヘッダーは単色の色帯（知識＝青、行動＝オレンジ）、追加ボタンはカード下部中央のピル型ボタンという構成。 元の実装はリスト（<ul>）で追加ボタンが右上にあったため、構造ごと差し替えた。

削除確認モーダルの再設計

プロフィール画面の退会確認モーダル（issue #16）で作ったパターン （ヘッダー＋警告ボックス＋フッター）を踏襲し、削除対象（タイトル・著者）と 「参照元になっている知識・行動も削除される」旨の注意書きを追加した。
