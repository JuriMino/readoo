Readoo! 結合テスト仕様書

対象バージョン：v1（本・知識・行動 管理機能 一式）

作成日：2026-07-12

0. この仕様書について

Readoo! v1 の全機能（認証／プロフィール／Book／Knowledge／Action）を、実画面を通して一連の流れで確認するための手動結合テスト仕様書。自動テスト（Feature Test）で保証済みのロジックも、実際のUI上での挙動として最終確認する位置づけ。

凡例

共通・認証 プロフィール Book Knowledge Action 権限・削除・整合性
  • 必須＝ 作り込んだ守り（認可・削除連鎖・整合性）で、特に確認価値が高い項目
  • 判定欄：合格なら OK、不具合なら NG を記入し、NG は memo に事象を残す
事前準備
  • テストユーザー2名を用意：userA（主役）と userB（他人データ検証用）
  • 各ユーザーで、本・知識・行動を最低1件ずつ登録できる状態にしておく
  • ブラウザ（ログイン状態を切り替えるため、シークレットウィンドウ併用が便利）
  • 可能ならDBを直接確認できる手段（退会時の物理削除確認用。無ければ画面挙動で代替）

AUTHZ-02他人の編集・更新・削除不可 必須userB で userA のデータの編集URL/更新/削除を試みる403、かつ userA のデータは変化しないOK / NG
AUTHZ-03一覧に他人のデータが出ない 必須userB で本/知識/行動の各一覧を開くuserA のデータが表示されない（自分の分だけ）OK / NG
AUTHZ-04他人の本に作成不可userB が userA の本を指定して知識/行動を作成しようとする（フォーム改ざん相当）book_idのバリデーションで弾かれ、作成されないOK / NG

8. 削除連鎖・整合性（DEL / VAL）

ID確認内容手順期待結果判定
DEL-01本削除で子データが隠れる 必須知識・行動を持つ本を削除し、知識一覧・行動一覧を開く／該当詳細URLを開く一覧から消え、詳細URLは403（論理削除で非表示。物理削除ではない）OK / NG
DEL-02退会で子データが物理削除 必須本・知識・行動を持つユーザーで退会するユーザーは論理削除され、本・知識・行動は物理削除（DB確認 or 再ログイン不可で代替）OK / NG
VAL-01別の本の知識は紐付け不可 必須本1の行動登録で、本2に属する知識を紐付けて送信（改ざん相当）knowledge_idのバリデーションで弾かれる（memo：通常UIではAlpineの絞り込みで選べない。自動テスト ActionAuthorizationTest でも保証済）OK / NG
実施のコツ：
  • 権限系（7章）は、userA でデータを作った後 そのURLを控えておき、userB のシークレットウィンドウで開くと効率的。
  • DEL-02（退会の物理削除）は画面だけだと分かりにくい。DBを見られるなら knowledges/actions テーブルが空になることを直接確認。難しければ「退会後にそのアカウントでログインできない＝退会成立」までで妥協可。
  • VAL-01は手動再現が難しいので、memo扱い＋自動テストで担保という記録で十分。
  • NGが出たら、memo欄に「どの画面で／何をしたら／どうなった」を残すと修正時に追いやすい。
